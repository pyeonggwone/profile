## 2) 예외/충돌/권한/race condition 등 분석

### A. 파일 누락의 직접 원인

- PR #2, #3, #4는 모두 머지되었지만, 각 PR의 `get_files` 결과가 `[]`(변경 파일 없음)으로 확인됨.
- 실제로 현재 저장소에도 `Prosci/README.md`, `Prosci/타임라인.md`가 존재하지 않음.
- 결론: **"파일 생성 커밋이 올라가기 전에 PR이 머지"**되어 파일 누락 발생.

### B. 런 취소와 머지 타이밍 충돌 (race condition)

- 로그상 `Progress reported successfully.` 직후 `The operation was canceled.`가 반복.
- 여러 요청/런이 짧은 간격으로 연속 생성되며, 이전 런이 취소되는 패턴 확인.
- PR이 체크 pending 상태에서 빠르게 머지된 기록 존재(로그에 `1 check was pending` 단서).
- 결론: **Copilot 작업 완료/커밋 반영 이전에 PR 머지 또는 새 런 트리거가 발생하는 타이밍 경합**.

### C. 권한/가시성 이슈

- PR check-runs 조회 시 API 응답:
  - `403 Resource not accessible by integration`
- 결론: 일부 체크 상세 조회 권한이 제한되어, 실패 원인 추적이 로그 기반으로만 가능.

### D. 워크플로우/코드 레벨 특이사항

- 이 저장소에는 `.github/workflows/*.yml`이 직접 존재하지 않으며, 실행 경로가 `dynamic/copilot-swe-agent/copilot`로 표시됨.
- 즉, 저장소 내 스크립트/워크플로우 코드를 직접 수정해 원인 제거하기 어려운 구조.

## 3) 재현 방법, Root Cause, 개선점, 유발 조건

### 재현 방법

1. Copilot에 동일/유사 요청을 짧은 간격으로 반복 전송
2. 생성된 PR에서 체크 완료 전(또는 에이전트 커밋 전) 즉시 머지
3. 이전 런이 취소되면서 `Processing Request` 단계에서 중단
4. 머지된 PR에 실질 변경 파일이 없는 상태 재현

### Root Cause (핵심)

1. **프로세스 문제(1순위)**: 에이전트가 실제 파일 생성/커밋하기 전에 PR이 머지됨
2. **동시성 문제(2순위)**: 연속 트리거로 이전 런이 자동 취소되는 경합
3. **관측성 문제(3순위)**: 일부 체크 상세 조회 403으로 세부 진단 제약

### 현상 유발 조건

- 체크가 pending인 상태에서 빠른 수동 머지
- 같은 목적의 요청을 연속 제출해 런 중첩
- 에이전트가 `Processing Request` 단계에 있을 때 새 이벤트 유입

### 개선점 (실행 우선순위)

1. **머지 게이트 강제**: `copilot` 체크 성공 전 머지 금지 (branch protection)
2. **중복 실행 제어**: 동일 대상 브랜치/PR에 대한 동시 실행 제한(concurrency cancel 정책 정교화)
3. **완료 신호 표준화**: PR 본문/코멘트에 "파일 생성 완료 + 변경 파일 목록" 확인 후 머지
4. **검증 자동화**: PR 머지 전 필수 파일 존재 검증(`Prosci/README.md`, `Prosci/타임라인.md`)
5. **권한 정비**: 체크런 조회 가능한 토큰/앱 권한 확장하여 장애 분석 정확도 개선

## 4) 기타 누락 기록/관련 현상

- PR #2~#4는 제목상 Prosci 파일 생성을 목표로 했지만, 실제 머지 결과물은 파일 생성이 반영되지 않음.
- 현재 브랜치/저장소 상태 기준으로 Prosci 산출물이 비어 있어, 사용자 체감상 "강제취소/누락"으로 보이는 것이 정상.
- 동일 패턴이 반복되었으므로 단발성 오류보다 **운영 절차(머지 타이밍 + 동시 실행 관리)** 이슈 성격이 큼.
