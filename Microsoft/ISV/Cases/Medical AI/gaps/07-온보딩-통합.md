# 온보딩 / 고객 통합 — 부족한 디테일 상세

## 현황 요약
현재 문서에는 "K3S 클러스터 추가 또는 VNet Peering 연결"로 신규 병원을 온보딩한다고만 나와 있다.
실제 세일즈→계약→기술 구성→운영 이관까지의 단계별 절차와
고객사 시스템과의 연동 API 명세가 전혀 없다.

---

## 1. 신규 병원 온보딩 절차 없음

### 현재
"K3S 엣지 클러스터 추가 또는 VNet Peering 연결"만 언급

### 전체 온보딩 프로세스 정의 필요

```
[Phase 0 - 사전 검토 (1~2주)]
□ 병원 IT 환경 조사
  - 기존 EMR 시스템 벤더 및 버전
  - 기존 PACS 시스템 연동 가능 여부
  - 온프레미스 서버 스펙 (Windows Server 버전, Docker 지원 여부)
  - 인터넷 회선 대역폭 및 방화벽 설정 가능 범위
  - 의료기기 규정 내 소프트웨어 설치 허용 여부
□ 규모 분류
  - 대형 병원: Hospital A 유형 (전용 Apache Flume + Docker 에이전트)
  - 중소 병원: Hospital B/C/D 유형 (K3S 클러스터)

[Phase 1 - 계약 및 환경 준비 (1주)]
□ 서비스 계약 체결 (SLA 포함)
□ 병원 전용 Subscription Set 프로비저닝 결정
  - 전용: 대형 병원 또는 규정상 데이터 격리 필요
  - 공유: 중소 병원
□ Azure 리소스 프로비저닝 (IaC 파이프라인 실행)
  - Subscription Set (AI + SW) 생성
  - VPN Gateway 또는 VNet Peering 설정
  - 병원 전용 DB 파티션 또는 인스턴스 생성
□ 온프레미스 에이전트 패키지 준비
  - Docker 이미지 버전 확정 (병원 OS 환경 맞춤)
  - 설치 스크립트 및 설정 파일 생성

[Phase 2 - 기술 구성 (1~2주)]
□ 온프레미스 에이전트 설치
  - Windows Server에 Docker CE 설치 및 검증
  - 에이전트 Docker Compose 파일 배포
  - PII Masking Container 설정 (병원 EMR 필드 매핑)
□ 네트워크 연결 구성
  - VPN 터널 설정 (병원 방화벽 팀 협력 필요)
  - 테스트 패킷 전송 확인
  - Private DNS 설정 검증
□ Azure Arc 등록
  - 온프레미스 VM → Azure Arc 연결
  - 헬스 모니터링 확인

[Phase 3 - 통합 테스트 (1주)]
□ 샘플 ECG 데이터 전송 (비식별 테스트 데이터)
□ 비식별화 처리 결과 검증 (PHI 잔여 여부 확인)
□ AI 분석 결과 반환 확인 (응답시간 측정)
□ 포털 화면에서 결과 조회 확인
□ 알림 기능 테스트 (분석 완료 → 병원 담당자 알림)
□ RabbitMQ DLQ 처리 시나리오 테스트

[Phase 4 - 운영 이관 (1주)]
□ 병원 담당자 사용자 교육
  - 포털 사용법
  - 에이전트 재시작 방법
  - 오류 발생 시 연락처
□ 운영팀 인수인계
  - 모니터링 대시보드 공유 (Grafana)
  - 알림 채널 설정 (병원 담당자 연락처)
□ 온보딩 완료 체크리스트 서명

[온보딩 예상 총 소요 기간]
대형 병원: 5~7주
중소 병원 (K3S): 3~4주
```

## 2. Client System Integration API 명세 없음

### 문제
"Client System Integration을 통해 고객사 자체 시스템과 연동"만 언급되고
실제 API 포맷·인증 방식·응답 구조가 없다.

### 필요한 API 설계

```
[API 유형]
1. ECG 데이터 제출 API (Upload)
2. 분석 결과 조회 API (Query)
3. 분석 완료 웹훅 (Webhook)
4. 병원 관리 API (Admin)

[인증 방식]
- 기관 간 시스템 연동: API Key (Header: X-Api-Key) 또는 OAuth 2.0 Client Credentials
- 개별 사용자 포털: Entra ID OIDC 토큰

[API 예시 설계]
---
POST /api/v1/ecg/submit
설명: 비식별화된 ECG 데이터 제출
인증: API Key

Request:
{
  "hospital_id": "hosp-001",
  "patient_hash": "sha256-hash",   // 비식별 환자 식별자
  "ecg_format": "DICOM",           // DICOM | XML | PDF
  "acquired_at": "2026-04-20T10:30:00Z",
  "file_url": "https://...",       // Shared Access Signature URL
  "checksum": "sha256-of-file"
}

Response:
{
  "request_id": "uuid-v4",         // 추적용 ID
  "status": "accepted",
  "estimated_completion_seconds": 30
}

---
GET /api/v1/ecg/{request_id}/result
설명: 분석 결과 조회
인증: API Key

Response:
{
  "request_id": "uuid-v4",
  "status": "completed",           // pending | processing | completed | failed
  "analysis_result": {
    "rhythm": "Normal Sinus Rhythm",
    "abnormalities": [],
    "confidence_score": 0.97,
    "analyzed_at": "2026-04-20T10:30:28Z",
    "model_version": "1.3.2"
  }
}

---
POST {customer_webhook_url}
설명: 분석 완료 시 고객사 시스템으로 결과 Push
인증: HMAC-SHA256 서명 (Webhook Secret)

Payload:
{
  "event": "ecg.analysis.completed",
  "request_id": "uuid-v4",
  "hospital_id": "hosp-001",
  "result_url": "https://.../api/v1/ecg/{id}/result",
  "timestamp": "2026-04-20T10:30:28Z"
}
```

## 3. 병원별 설정(Configuration) 관리 없음

### 문제
각 병원마다 EMR 필드 매핑, 알림 설정, 접근 권한이 다를 텐데
이를 어디에 어떻게 저장·관리하는지 없다.

### 설정 관리 설계

```
[설정 저장소]
MySQL: hospital_config 테이블
{
  hospital_id       : "hosp-001",
  display_name      : "서울대학교병원",
  tier              : "dedicated",        // dedicated | shared
  subscription_set  : "set-1",
  pii_field_mapping : { ... },            // 병원별 EMR 필드 → 표준 필드 매핑
  webhook_url       : "https://...",      // 분석 완료 알림 엔드포인트
  webhook_secret    : "(Key Vault 참조)", // 직접 저장 금지
  notification_email: "admin@hospital.kr",
  max_concurrent_requests: 10,
  data_retention_years: 10,
  created_at        : "2026-01-01",
  status            : "active"
}

[설정 변경 프로세스]
1. 병원 담당자 → 포털에서 변경 요청
2. MedicalAI 운영팀 → 검토 및 승인
3. 설정 DB 업데이트 + 변경 이력 감사 로그 기록
4. 온프레미스 에이전트 설정 갱신 (Azure Arc Policy or 재배포)

[비밀키 관리]
- webhook_secret, API Key: MySQL 직접 저장 금지
- Key Vault에 저장 → MySQL에는 Key Vault 시크릿 참조 URL만 보관
```

## 4. 계약 종료 / 오프보딩 절차 없음

### 필요한 오프보딩 절차

```
[계약 종료 시 처리 순서]
1. 데이터 반환 (30일 이내)
   - 병원 요청 데이터: 암호화된 형태로 병원 지정 서버로 전송
   - 전송 방법: Azure Data Factory → 병원 제공 SFTP 또는 S3

2. 데이터 파기 (반환 완료 후)
   - CosmosDB, MySQL에서 hospital_id 기반 데이터 삭제
   - Storage Account 관련 Blob 삭제
   - 파기 방법: NIST SP 800-88 기준 (덮어쓰기 후 삭제)
   - 파기 확인서 발급

3. 인프라 해제
   - 전용 Subscription Set: 리소스 삭제 후 구독 해지
   - VPN Gateway 연결 해제
   - Azure Arc 연결 해제

4. 접근 권한 제거
   - 병원 관련 Entra ID 계정 비활성화
   - API Key 즉시 폐기

5. 오프보딩 완료 보고서 발행
```

## 5. 고객 포털 UX / 기능 정의 없음

### 현재
"최종 분석 결과는 사용자가 웹 페이지에서 확인"만 언급

### 포털 기능 정의 필요

```
[병원 사용자 포털 기능]
1. ECG 제출 현황
   - 제출 일시, 처리 상태 (대기/처리중/완료/실패), 처리 시간
2. 분석 결과 조회
   - ECG 파형 시각화 + AI 분석 결과 텍스트
   - 결과 PDF 출력/다운로드
3. 통계 대시보드
   - 월별 제출 건수, 이상 소견 비율 추이
4. 설정 관리
   - 알림 이메일 설정, Webhook 설정
5. 사용량 조회
   - 월별 분석 건수 및 청구 예정 금액

[접근 권한]
- 병원 관리자: 전체 기능
- 의사/의료진: 결과 조회 (자기 환자 범위)
- 청구 담당자: 사용량/청구 조회만
```
