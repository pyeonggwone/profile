# 데이터 파이프라인 세부 — 부족한 디테일 상세

## 현황 요약
현재 문서에서는 "PII Masking Container(Python, Java)"를 통해 비식별화 후
RabbitMQ로 전송하는 흐름이 있으나,
비식별화 기법, 데이터 품질 검증, MongoDB 인덱싱, 보존 정책 등 세부가 없다.

---

## 1. PII Masking / 비식별화 기법 미정의

### 문제
어떤 방법으로 비식별화하는지 명시가 없으면 규정 준수 여부를 증명할 수 없다.

### 의료 데이터 비식별화 기준 (보건복지부 가이드라인)

```
[식별 가능 항목 (PHI — Protected Health Information)]
직접 식별자 (제거 필수)
  - 이름, 주민등록번호, 전화번호, 이메일
  - 주소 (시/구 이하)
  - 계좌번호, 의무기록번호(MRN)
  - 의료기기 일련번호

준식별자 (변환 또는 일반화)
  - 생년월일 → 출생연도만 유지 또는 5세 구간 일반화
  - 특정 진단코드 + 희귀질환 → 상위 카테고리로 변환
  - 상세 주소 → 행정구역 수준만 유지

[비식별화 기법 선택]
기법              | 설명                          | ECG 데이터 적용 여부
─────────────────|───────────────────────────── |──────────────
가명처리          | 이름→가명 치환 (역추적 키 존재) | 병원 내부 처리용
익명화            | 완전 제거 (역추적 불가)         | 클라우드 전송 데이터
k-익명성          | k명 이상 동일 속성 보장         | 선택적 적용
데이터 마스킹     | 특정 필드 **** 치환             | 로그·화면 표시용
```

### PII Masking Container 구현 상세 (정의 필요)

```python
# 처리 항목 예시 (현재 코드 명세 없음)

ECG XML/DICOM 헤더에서 제거 대상 태그:
  DICOM Tag (0010,0010): Patient Name → "ANONYMOUS"로 대체
  DICOM Tag (0010,0020): Patient ID → SHA-256 해시값으로 대체
  DICOM Tag (0010,0030): Birth Date → 연도만 유지 (YYYY0101)
  DICOM Tag (0010,0040): Patient Sex → 유지 (의학적 의미 있음)
  DICOM Tag (0008,0022): Acquisition Date → 유지 (ECG 시간 정보 필요)

처리 후 검증:
  - 잔여 PHI 스캔 (정규식 기반 잔여 PII 탐지)
  - 처리 결과 메타데이터 로그 기록 (어떤 필드를 처리했는지)
  - 처리 실패 시: 전송 차단 + 알림 발송
```

## 2. 데이터 품질 검증(Validation) 없음

### 문제
수집된 ECG 데이터의 유효성을 검사하지 않으면
노이즈 데이터가 AI 분석 모듈에 입력되어 오진 위험이 발생한다.

### 필요한 검증 단계

```
[수집 단계 검증 — 온프레미스]
1. 파일 형식 검증
   - 허용 포맷: XML, DICOM, PDF, XML+IMAGE
   - 파일 크기 범위: 최소 N KB ~ 최대 N MB (정의 필요)
   - 파일 무결성: SHA-256 체크섬 전송 전 계산 → 클라우드에서 확인

2. 필수 필드 검증
   - ECG 리드 수: 표준 12리드 여부 확인
   - 샘플링 레이트: 허용 범위 (500Hz, 1000Hz 등)
   - 측정 시간: 최소 10초 이상 여부
   - 환자 연령 범위 유효성 (출생연도 기준)

[클라우드 수신 단계 검증 — AKS 마이크로서비스]
3. 비식별화 완료 확인
   - PHI 잔여 스캔 (이중 검증)
   - 체크섬 검증

4. 중복 데이터 감지
   - 동일 ECG ID + 동일 병원 + 동일 시간대 중복 제출 감지
   - 중복 시: 최신 데이터 우선, 이전 버전 Soft Delete

[검증 실패 처리]
  - 실패 건: Dead Letter Queue(DLQ)로 이동
  - 알림: 송신 병원 에이전트에 오류 코드 반환
  - 재처리: 운영팀 수동 검토 후 재전송 또는 폐기
```

## 3. MongoDB (CosmosDB) 인덱싱 전략 없음

### 문제
ECG 데이터 조회 패턴에 맞는 인덱스가 없으면 쿼리 응답이 느려지고
비용이 RU(Request Unit) 낭비로 이어진다.

### 예상 주요 쿼리 패턴 및 인덱스 설계

```javascript
// 컬렉션: ecg_records

// 주요 조회 패턴
1. 병원 + 날짜 범위로 ECG 목록 조회
2. 환자 가명 ID로 ECG 이력 조회
3. AI 미분석 데이터 배치 조회 (분석 대기열)
4. 분석 완료 데이터 기간별 통계 집계

// 권고 인덱스 설계
{
  "hospital_id": 1,           // 병원 필터
  "acquired_at": -1           // 날짜 내림차순 정렬
}  // → 패턴 1 커버

{
  "patient_hash": 1,          // 가명 환자 ID
  "acquired_at": -1
}  // → 패턴 2 커버

{
  "analysis_status": 1,       // "pending" | "processing" | "done"
  "created_at": 1
}  // → 패턴 3 커버 (부분 인덱스: status="pending"만)

// CosmosDB 파티션 키 선택
- 권고: hospital_id (병원 단위로 데이터 분산)
- 주의: 단일 병원 데이터 집중 시 Hot Partition 발생 가능
  → 대형 병원은 hospital_id + 월(YYYY-MM) 복합 파티션 검토
```

## 4. 데이터 보존 정책 /  파기 절차 없음

### 법적 보존 기준

| 데이터 종류 | 법적 보존 기간 | 적용 법령 |
|-------------|----------------|-----------|
| 진료기록 (ECG 원본) | 10년 | 의료법 제22조 |
| 처방전 | 2년 | 약사법 |
| 영상기록 (DICOM) | 5년 | 의료법 시행규칙 |
| 서비스 이용 로그 | 3년 | 개인정보보호법 |
| 결제 정보 | 5년 | 전자상거래법 |

### 데이터 생명주기 관리

```
[CosmosDB — ECG 분석 데이터]
- 보존: 병원 계약 기간 + 5년 (최소)
- 계약 종료 시: 30일 내 안전 파기 (NIST SP 800-88 기준)
- TTL(Time to Live): CosmosDB TTL 기능으로 만료 데이터 자동 삭제 설정

[MySQL — 서비스 운영 데이터]
- 결제 정보: 5년 보관 후 일괄 삭제
- 사용자 계정: 탈퇴 후 30일 보관 → 파기

[스토리지 — AI 분석 결과]
- Storage Account: Hot → Cool (90일) → Archive (1년) → 삭제 (보존 기간 초과)
- Azure Lifecycle Management 정책으로 자동 이동

[파기 확인서]
- 병원 요청 시 데이터 파기 완료 확인서 발급 절차 필요
```

## 5. RabbitMQ 메시지 처리 세부 없음

### 큐 설계 및 메시지 스키마

```
[큐 구조]
ecg.raw.realtime     # 실시간 스트리밍 ECG 데이터
ecg.raw.batch        # 배치 파일 ECG 데이터
ecg.processed        # 전처리 완료 → AI 분석 대기
ecg.analysis.result  # AI 분석 완료 결과
ecg.dlq              # Dead Letter Queue (처리 실패 메시지)

[메시지 스키마 (미정의)]
{
  "message_id": "uuid-v4",
  "hospital_id": "hosp-001",
  "patient_hash": "sha256-hash",
  "ecg_format": "DICOM | XML | PDF",
  "file_size_bytes": 1024000,
  "checksum": "sha256-of-file",
  "acquired_at": "2026-04-20T10:30:00Z",
  "submitted_at": "2026-04-20T10:31:00Z",
  "metadata": {}
}

[재처리 정책]
- 실패 시 재시도: 최대 3회 (지수 백오프: 1초, 5초, 30초)
- 3회 모두 실패: DLQ로 이동 + 운영팀 알림
- DLQ 보관: 최대 7일 → 수동 처리 후 삭제 또는 재전송

[메시지 처리 순서 보장]
- 동일 환자의 ECG 처리 순서 보장 필요 여부 검토
  → 필요 시: Consistent Hash Exchange 또는 단일 Consumer 할당
```

## 6. 멀티클라우드 연동 데이터 흐름 세부 없음

### AWS DataSync 연동

```
[현재]: "AWS DataSync를 통해 AWS 환경 데이터 동기화"만 언급

[필요한 세부]
- AWS 소스: S3 버킷 또는 EFS
- Azure 대상: Azure Blob Storage (Shared Subscription)
- 동기화 스케줄: 실시간 vs 1시간 주기 배치
- 비식별화 처리 시점: AWS 측에서 처리 후 전송 vs Azure 수신 후 처리
- 네트워크 경로: AWS Direct Connect + Azure ExpressRoute 또는 인터넷 VPN
- 비용: DataSync 전송량 기준 과금 → 예산 설정 필요
```

### Azure Data Factory 연동

```
[현재]: "GCP, RDBMS 등 이기종 소스 연동"만 언급

[필요한 세부]
- GCP Cloud Storage → Azure Blob: Managed Integration Runtime 사용
- RDBMS(Oracle, PostgreSQL 등): Self-Hosted IR 배포 필요
  → 고객사 온프레미스에 ADF Self-Hosted IR 설치
- 파이프라인 실행 스케줄: 야간 배치 (04:00 AM KST)
- 실패 시 알림: ADF 모니터링 → Azure Monitor 연동
```
