# AI 모델 관리 (MLOps) — 부족한 디테일 상세

## 현황 요약
현재 문서에는 "AI Analysis Module이 ECG 데이터를 분석"하고 "Azure GPU 리소스로 학습"한다는 언급만 있다.
모델이 어떻게 개발되고, 검증되고, 배포되고, 운영 중에 관리되는지 전혀 정의되어 있지 않다.

---

## 1. 모델 개발 및 실험 관리 없음

### 문제
여러 연구자/개발자가 ECG 모델을 실험할 때 재현성과 비교가 불가능하다.

### 필요한 구성: Azure Machine Learning Workspace

```
[실험 추적 (Experiment Tracking)]
- 도구: Azure ML Workspace + MLflow 통합
- 기록 항목:
    • 학습 데이터 버전 (CosmosDB 스냅샷 ID 또는 Azure ML Dataset 버전)
    • 하이퍼파라미터 (학습률, 배치 크기, 에폭 수 등)
    • 평가 지표 (Accuracy, AUC-ROC, F1-Score, Sensitivity, Specificity)
    • 학습 소요 시간, GPU 사용량
    • 환경 정보 (Python 버전, 라이브러리 버전)

[재현성 보장]
- 학습 코드: Git 태그로 버전 고정
- 학습 데이터: Azure ML Dataset 버전 스냅샷
- 환경: Docker 이미지 버전 고정 (ACR에 저장)
```

## 2. 모델 학습 파이프라인 없음

### 현재
"Azure GPU 리소스로 학습"만 언급, 학습 트리거·주기·데이터 준비 과정 없음

### 필요한 파이프라인 설계

```
[학습 파이프라인 단계]

Step 1: 학습 데이터 준비
  - CosmosDB에서 비식별 ECG 데이터 쿼리
  - 데이터 품질 필터링 (노이즈, 결측값 제거)
  - Train / Validation / Test 분리 (70:15:15)
  - Azure ML Dataset으로 버전 등록

Step 2: 전처리
  - ECG 신호 정규화, 노이즈 필터링
  - 분류 레이블 매핑 (진단 코드 → AI 목표값)
  - 데이터 증강 (Data Augmentation) 적용 여부 결정

Step 3: 모델 학습
  - Azure ML Compute Cluster (GPU: NCv3 또는 ND 시리즈)
  - 분산 학습 여부: 단일 GPU vs 멀티 GPU
  - 재학습 트리거: 수동 요청 또는 특정 조건 자동 트리거

Step 4: 모델 평가
  - Test Set 기준 성능 지표 자동 계산
  - 이전 배포 모델과 성능 비교 (Challenger vs Champion)
  - 임계값(Threshold) 미달 시 배포 차단

Step 5: 모델 등록 및 패키징
  - Azure ML Model Registry에 버전 등록
  - ONNX 또는 TensorRT 포맷으로 최적화 변환
  - Docker 이미지 빌드 → ACR 푸시

Step 6: 배포
  - ACA(Azure Container Apps) AI 분석 모듈 업데이트
  - Blue/Green 배포로 무중단 전환
```

## 3. 모델 버전 관리 전략 없음

### 필요한 버전 관리 체계

```
[Azure ML Model Registry 구조]
모델명: ecg-classifier
버전 체계: <major>.<minor>.<patch>
  - Major: 모델 아키텍처 변경 (ResNet → Transformer 등)
  - Minor: 학습 데이터 추가 또는 하이퍼파라미터 조정
  - Patch: 버그 수정 또는 후처리 로직 변경

[스테이지 관리]
Development → Staging → Production
  - Development: 실험 단계, 미검증
  - Staging: 성능 검증 완료, 임상 검토 대기
  - Production: 실제 서비스 중

[롤백 정책]
- Production 모델 이전 버전을 최소 3개 보관
- 배포 후 성능 저하 감지 시 1-Click 이전 버전으로 롤백
- 롤백 기준: 주요 지표 5% 이상 하락
```

## 4. 재학습(Retraining) 트리거 없음

### 필요한 트리거 기준

| 트리거 유형 | 조건 | 대응 |
|-------------|------|------|
| **시간 기반** | 분기별 1회 | 정기 재학습 스케줄 |
| **데이터 기반** | 신규 레이블 데이터 N건 누적 | 자동 재학습 파이프라인 시작 |
| **성능 기반** | 드리프트 감지 알림 발생 | 긴급 재학습 검토 |
| **수동** | 임상팀 또는 AI팀 요청 | 즉시 재학습 실행 |

## 5. 모델 성능 모니터링 없음

### 문제
배포 이후 모델이 실제 환경에서 어떻게 동작하는지 추적하지 못한다.

### 필요한 모니터링 체계

**데이터 드리프트(Data Drift) 감지**
```
- 입력 ECG 데이터의 통계적 분포 변화 감지
  예: 특정 병원에서 수집한 리드 수 변경, 샘플링 레이트 변경
- 주기: 주 1회 자동 검사 (Azure ML Data Drift Monitor)
- 임계값 초과 시: AI팀 알림 → 재학습 여부 검토
```

**예측 성능 모니터링**
```
- 방법: 생산 데이터 중 일부에 대해 임상의 검토 결과와 AI 결과 비교
- 지표: Positive Predictive Value (PPV), Sensitivity
- 주기: 월 1회 샘플링 검토 (100건 이상)
- 보고: AI 성능 월간 리포트 → 임상팀 + 개발팀 공유
```

**추론 지연(Latency) 모니터링**
```
- 목표: ECG 1건 분석 응답시간 ≤ N초 (미정의 → 정의 필요)
- 측정: ACA → Application Insights 통합
- P50 / P95 / P99 분위수 추적
- P99 > 임계값 시 자동 스케일 아웃 트리거
```

## 7. AI 결과의 임상 검증 프로세스 없음

### 문제
AI 분석 결과가 의료 행위에 참고될 경우, 임상적 안전성 검증 절차가 필요하다.

### 필요한 프로세스

```
[신규 모델 배포 전 임상 검증]
1. 내부 검증 (AI팀): Test Set 성능 지표 검토
2. 임상 검증 (의료 자문): Shadow Mode 배포
   - 실제 서비스에서 기존 모델과 신규 모델 동시 추론
   - 결과는 신규 모델 결과를 사용자에게 미표시
   - 임상의가 양 모델 결과 비교 후 승인
3. 단계적 배포 (Canary): 10% → 30% → 100% 트래픽 전환
4. 모니터링 기간: 배포 후 2주 집중 관찰

[SaMD 허가 관련]
- AI 분석 결과가 진단 보조에 해당하는 경우 식약처 허가 필요
- 허가 전: 결과 화면에 "의료 전문가 확인 필수" 고지 필수
```
