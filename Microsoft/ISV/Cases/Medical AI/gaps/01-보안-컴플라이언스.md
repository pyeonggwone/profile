# 보안 / 컴플라이언스 — 부족한 디테일 상세

## 현황 요약
현재 문서에는 "Defense in Depth" 원칙과 PII Masking, Private Endpoint, Key Vault 사용이 명시되어 있으나,
각각의 구체적 구현 방법 및 규정 준수 근거가 없다.

---

## 1. 규정 준수 (Compliance) 기준 미정의

### 문제
플랫폼이 어느 규정을 따르는지 명시되어 있지 않다.

### 확인이 필요한 항목

| 규정 | 적용 대상 | 주요 요구사항 |
|------|-----------|---------------|
| **의료법 제21조** | 국내 병원 데이터 | 진료기록 보존 10년, 열람 기록 유지 |
| **개인정보보호법** | 환자 PII | 비식별화 기준, 동의 관리, 파기 절차 |
| **의료기기법** | AI 분석 모듈 | 소프트웨어 의료기기(SaMD) 허가 여부 |
| **ISMS-P** | 정보보호 관리체계 | 인증 취득 여부 |
| **HIPAA** | 해외 병원 연동 시 | PHI 처리·저장 기준 (해외 진출 시 필수) |

### 즉시 확인할 질문
- AI 분석 결과물이 의료 행위 보조에 해당하는가? → SaMD 허가 대상 여부 결정
- 해외 병원(미국, 유럽) 온보딩 계획이 있는가? → HIPAA / GDPR 적용 범위 결정
- Azure 리전이 Korea Central 단독인가? → 데이터 국외 이전 여부 판단

## 2. IAM (ID 및 접근 관리) 설계 없음

### 문제
누가 어떤 리소스에 접근하는지, 어떤 방식으로 인증하는지 정의가 없다.

### 필요한 설계 요소

**Microsoft Entra ID 연동**
```
[적용 대상]
- MedicalAI 내부 개발자/운영자 → Entra ID 디렉터리 계정
- 병원 관리자 → B2B Guest 초대 또는 외부 IdP 연동 (SAML/OIDC)
- 서비스 간 인증 → Managed Identity (Workload Identity)

[다층 인증]
- 운영자 Privileged Account → MFA 필수 + Conditional Access
- 병원 포털 사용자 → MFA 권장
- 온프레미스 에이전트 → 서비스 주체(Service Principal) + 인증서 기반
```

**RBAC 역할 설계 (최소 권한 원칙)**
```
역할               | 권한 범위
───────────────────|─────────────────────────────────────────
플랫폼관리자       | Shared/Hub Subscription 전체 관리
AI운영자           | AI Subscription 읽기 + AI 모듈 재배포
SW개발자           | SW Subscription 배포 권한 (Key Vault 외)
병원관리자         | 자사 Subscription Set의 모니터링 조회만
데이터분석가       | CosmosDB 읽기 전용 (비식별 데이터만)
보안감사자         | 모든 구독 로그 읽기 전용
```

**Managed Identity 적용 위치**
- AKS → Key Vault 접근: Workload Identity 사용
- AKS → CosmosDB/MySQL 접근: Managed Identity 기반 연결 문자열
- Azure Data Factory → 원본 데이터: Managed Identity (AWS 제외)

## 3. 네트워크 보안 세부 설계 없음

### NSG(Network Security Group) 규칙

```
[SW Subscription VNet]
인바운드
- AKS → MySQL: 3306 허용 (AKS 서브넷 소스)
- AKS → CosmosDB: Private Endpoint 경유 전용
- Load Balancer → AKS: 443 허용
- Hub VPN GW → SW VNet: VPN 트래픽 허용
아웃바운드
- 인터넷 직접 아웃바운드: 차단 (Hub 경유 강제)

[AI Subscription VNet]
인바운드
- SW Subscription AKS → ACA Private Endpoint: 443 허용
- 그 외 인바운드: 전체 차단
```

### Azure Firewall / DDoS
- Hub Subscription에 **Azure Firewall Premium** 배치 권고
  - TLS 검사(IDPS) 활성화
  - FQDN 기반 아웃바운드 필터링
- 공개 엔드포인트가 있는 경우 **Azure DDoS Protection Standard** 적용

### 허용 포트 목록 (정의 필요)

| 출발지 | 목적지 | 포트 | 프로토콜 | 용도 |
|--------|--------|------|----------|------|
| 병원 온프레미스 | Hub VPN GW | 500, 4500 | UDP | IPsec VPN |
| Hospital A Docker | LB | 443 | HTTPS | 데이터 전송 |
| AKS Pod | Key Vault | 443 | HTTPS | 비밀키 조회 |
| AKS Pod | CosmosDB PE | 10255 | TCP | MongoDB API |
| ACA | AI 추론 모듈 | 443 | HTTPS | 내부 API 호출 |

## 4. 감사 로그 (Audit Log) 체계 없음

### 문제
의료 데이터에 대한 접근 이력 추적이 명시되어 있지 않다.
규제 기관 조사 시 6개월~2년 치 로그를 제출해야 하는 경우가 발생한다.

### 필요한 로그 항목

```
[데이터 접근 로그]
- 누가(사용자/서비스) 어떤 ECG 레코드를 조회/수정/삭제했는가
- 언제, 어느 IP에서 접근했는가
- 정상 접근인가, 이상 접근인가 (Azure Sentinel 연동)

[로그 저장 전략]
- Azure Monitor Diagnostic Settings → Log Analytics Workspace
- 보존 기간: 90일 Hot + 2년 Archive (Storage Account Cold Tier)
- 로그 변조 방지: Immutable Storage (Blob WORM) 적용

[Sentinel 연동]
- 비정상 접근 패턴 탐지 (동일 계정 단시간 대량 조회)
- 온프레미스 에이전트 이상 동작 탐지
- 자동 인시던트 생성 → 보안팀 알림
```

## 5. 비밀키 / 인증서 관리 세부 없음

### Key Vault 운영 기준 (현재 "사용"만 언급됨)

```
[회전(Rotation) 정책]
- DB 연결 문자열: 90일마다 자동 회전
- AKS 서비스 인증서: 6개월마다 자동 갱신 (cert-manager 연동)
- VPN PSK: 연 1회 수동 갱신

[접근 로그]
- Key Vault Diagnostic Settings → Log Analytics 전송
- 비인가 접근 시도 → Sentinel 경보

[비상 접근 (Break Glass)]
- 주 계정 잠금 시를 대비한 비상 관리자 계정 별도 관리
- 비상 계정 사용 시 즉시 알림 발송
```
