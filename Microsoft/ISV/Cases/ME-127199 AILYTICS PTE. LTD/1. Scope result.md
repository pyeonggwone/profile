2. Situation — Problem
파트너가 해결하고자 하는 핵심 문제를 다음과 같이 분해함.

2.1 AWS → Azure 마이그레이션 미완료
Root cause: Vision AI 워크로드가 AWS에 구축되어 있으며, Azure로의 service mapping(Compute/Storage/Networking/AI/Container)이 검증 중.
Impact: Azure 인스턴스가 production-ready가 되기 전까지 Marketplace 게시 불가 → 게시 일정이 May–June 2026으로 지연.
2.2 하이브리드(Edge + Cloud) 아키텍처와 Marketplace 요건 정합성 미확인
Root cause: 다수 고객이 데이터 민감성/규제로 인해 on-premises AI 추론을 요구. 카메라 스트림 처리 / AI 추론은 Edge, Alert·Insight·Web Dashboard는 Cloud로 분리.
Impact: SaaS와 Azure Managed Application 중 어떤 Offer 유형이 하이브리드 모델에 적합한지 확정되지 않음 → 게시 형태 결정 보류.
2.3 Tenant 생성 흐름 및 Production Readiness 미검증
Root cause: SaaS 게시 시 multi-tenant fulfillment(landing page + Subscription/Webhook API), Managed App 게시 시 ARM template + Managed Identity 구조에 대한 검증 부재.
Impact: 게시 후 Customer onboarding 자동화/격리/Billing 연동에서 장애 발생 위험.
2.4 Azure-native AI 서비스 채택 부재 → 프로그램 혜택/크레딧 미수령
Root cause: AI 기능이 자체 컨테이너 내부에 패키징되어 있어 Azure AI Foundry / Azure OpenAI / Azure AI Vision / Azure AI Content Understanding 등의 채택 포인트가 명시되지 않음.
Impact: ISV AI 관련 프로그램 (Pegasus, AI Sprint, Azure credits) 혜택 활용 어려움.
2.5 Pricing/Offer 구조 설계 미완
Root cause: Public(저가 Base) + Private(엔터프라이즈 번들) 전략은 정해졌으나 Plan/Meter/Private Plan/Custom Meter Dimensions 구체 설계 부재.
Impact: Transactable 설정 단계에서 Partner Center 등록 지연 위험.
3. Task — Suggestion
각 문제에 대응하는 실행 가능한 Microsoft 정렬 권고안.

3.1 AWS → Azure 서비스 매핑 가속화
Action: Azure Migrate 및 AWS-to-Azure service mapping 가이드를 활용하여 Compute/Storage/Networking/Container/AI 단위로 1:1 매핑 표 작성. Container 워크로드는 우선 Azure Kubernetes Service (AKS) 또는 Azure Container Apps 로 마이그레이션.
Why: 컨테이너 기반 Vision AI는 K8s 기반 운영이 표준이며, ARS에서 검증 가능한 Reference Architecture 확보 가능.
3.2 하이브리드 아키텍처 패턴 표준화
Action: Edge 추론은 Azure IoT Operations 또는 Azure Arc-enabled Kubernetes + Azure ML on Edge 패턴으로 정렬. Cloud 측은 Event Hubs → Stream Analytics/Functions → Cosmos DB → Web App/Front Door 기반 Alert/Dashboard 파이프라인으로 표준화.
Why: Microsoft가 공식 권고하는 Hybrid AI 패턴을 따르면 ARS 통과가 용이하고, Marketplace Offer 유형(특히 Managed App/Container Offer) 선택 근거가 명확해짐.
3.3 Offer 유형 결정 — SaaS + Azure Application(Managed App) 듀얼 게시 권고
Action:
SaaS Offer: 100% 클라우드 고객용 (Web Dashboard 중심). Multi-tenant fulfillment API 구현.
Azure Application (Managed App): 하이브리드 고객용. ARM template으로 고객 구독 내 리소스 자동 배포 + Managed Resource Group으로 운영 격리.
Why: 클라우드 전용/하이브리드 고객을 모두 커버하면서, Private Offer를 통한 엔터프라이즈 번들 판매를 양쪽 Offer 유형에서 모두 지원 가능.
3.4 Azure-native AI 서비스 통합 포인트 식별
Action:
영상 전처리/메타데이터 추출 → Azure AI Vision (Image Analysis 4.0, Spatial Analysis)
자연어 Insight/Alert Summary → Azure OpenAI Service (GPT-4o / GPT-4o-mini)
멀티모달 Insight → Azure AI Content Understanding (preview)
모델 운영 → Azure AI Foundry (Model Catalog, Deployments, Evaluations)
Why: Azure-native AI 채택 시 ISV AI 프로그램 혜택(크레딧, Co-sell, Pegasus) 활용이 가능하며, ARS에서 "Azure value-add" 항목을 충족.
3.5 Transactable 설정 및 Pricing 구조 설계
Action:
Base Public Plan (저가, Self-service onboarding)
Enterprise Private Plan (Software + Professional Services 번들, Custom Meter Dimension으로 카메라 스트림 수/시간 기준 과금)
Tax/Payout 프로파일 사전 완료 (HTME에서 합의된 항목)
Why: Microsoft가 Billing/Invoicing을 대행하므로 파트너의 청구 부담이 감소하고, Private Offer는 MACC (Microsoft Azure Consumption Commitment) 자격으로 엔터프라이즈 구매 확률을 높임.
