## Redis Usage and Cost Optimization

Redis was discussed as a cost and architecture review item because ELISA currently uses a Redis-compatible service and the camera scale is expected to grow from approximately 4,000 cameras toward 7,000 cameras. The decision is not only which Azure service to use, but whether a managed Redis-compatible option or a self-managed Redis-compatible deployment provides the right balance between cost, availability, operational ownership, and complexity.

| Area | Meeting-specific context | Review point |
|---|---|---|
| Current usage | Redis-compatible service in the existing platform | Confirm cache usage pattern, persistence needs, and peak load |
| Managed option | Azure-managed Redis-compatible cache | Evaluate operational simplicity, availability, monitoring, and pricing |
| Self-managed option | Redis-compatible deployment in containers or VMs | Evaluate potential cost savings against patching, backup, failover, and support ownership |
| Scale consideration | Growth from about 4,000 cameras toward 7,000 cameras | Test expected connection count, throughput, latency, and failover behavior |

Recommended review approach:

1. Confirm the current Redis-compatible workload pattern.
   - Cache-only or persistent data
   - Average and peak request rate
   - Data size and key expiration behavior
   - Availability and failover requirements
2. Compare managed and self-managed options.
   - Managed option: Azure Cache for Redis
   - Self-managed option: Redis OSS or Valkey on AKS or virtual machines
3. Run a cost and operations comparison.
   - Monthly infrastructure cost
   - Monitoring and alerting requirements
   - Backup and restore ownership
   - Patch and version upgrade ownership
   - Recovery time expectations
4. Validate under representative scale.
   - Current camera volume baseline
   - Expected year-end growth scenario
   - Latency and throughput under test traffic
   - Application behavior during restart or failover

## Multi-Tenant Architecture and Microsoft Entra ID

Multi-tenant architecture was discussed because ELISA currently operates under a single domain for all clients, while future plans include customer tenant ID registration and Microsoft Entra ID integration. Since cross-tenant synchronization is not applicable to the current single-domain setup, the immediate design question is how ELISA should define tenant boundaries and how Microsoft Entra ID should be introduced for future authentication and tenant-aware access.

| Area | Meeting-specific context | Review point |
|---|---|---|
| Current state | Single domain for all clients | No current tenant synchronization requirement |
| Future identity model | Microsoft Entra ID integration and tenant ID registration | Confirm whether Entra ID is for authentication only or also tenant administration |
| Tenant boundary | Boundary is not yet finalized | Decide whether the boundary is customer, project, deployment, site, or another model |
| Data isolation | Alerts, cameras, sites, users, and metadata need tenant-aware access | Confirm data filtering and authorization model before implementation |

Recommended review approach:

1. Define the tenant boundary model.
   - Customer-level boundary
   - Project-level boundary
   - Deployment-level boundary
   - Site-level boundary
2. Map core ELISA objects to the selected boundary.
   - Users
   - Cameras
   - Sites
   - Alerts
   - Metadata
   - Access roles
3. Plan Microsoft Entra ID integration.
   - Register ELISA as a multitenant application when external customer tenant sign-in is required
   - Capture and store the customer tenant ID during onboarding
   - Decide whether customer admins need delegated administration features
4. Validate tenant-aware access.
   - Confirm that users from one tenant boundary cannot access another tenant boundary's records
   - Test dashboard queries, alert views, camera lists, and metadata filters
   - Confirm the operational process for onboarding and offboarding customer tenants
