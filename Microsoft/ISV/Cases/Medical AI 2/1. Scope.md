## 2. Situation - Problem part

### Issue 1. HIPAA-aware on-premises deployment approach

Medical AI is reviewing an on-premises deployment approach in consideration of HIPAA requirements. The partner wants Microsoft guidance on practical best practices for hospital environments, especially where infrastructure policies and constraints may differ between domestic hospitals and US or other international hospitals.

Root cause: hospital infrastructure is not always standardized, and international hospitals may impose different operating system, security, network, and runtime constraints.

Impact: without a clear deployment baseline, the partner may face inconsistent installation experiences, longer customer onboarding, and additional validation effort for hospital security reviews.

### Issue 2. PowerShell-based installation versus container runtime introduction

The partner believes a PowerShell-based installer can be practical where introducing a container runtime is difficult. However, the partner's previous domestic hospital deployments usually used dedicated infrastructure rather than adapting to hospital-provided servers, so OS version inconsistency and compatibility have not been major issues so far.

Root cause: PowerShell can simplify Windows host automation, but using it as the main deployment model may diverge from the partner's current Linux-based cloud architecture.

Impact: a broad PowerShell-based model may increase long-term maintenance if cloud and on-premises architectures evolve separately.

### Issue 3. k3s-based deployment strategy

The current cloud service is Linux-based, and the partner is prioritizing a k3s-based on-premises deployment model to minimize architectural change while operating cloud and on-premises environments in parallel.

Root cause: the partner needs short-term architectural consistency and cannot wait for a dedicated on-premises architecture redesign.

Impact: k3s can reduce architecture drift, but the partner still needs a clear baseline for cluster prerequisites, image distribution, security controls, patching, monitoring, and operational ownership.

### Issue 4. Hybrid deployment for Windows-based components

Some components may need to run on Windows-based hosts, so the partner is considering selectively applying PowerShell-based installation while keeping the overall architecture aligned with the k3s model.

Root cause: not every hospital-side component can be assumed to run inside the same Linux container platform.

Impact: without a defined split between Kubernetes-managed components and Windows host-managed components, the deployment package may become difficult to support and validate.

## 3. Task - Suggestion part

### Recommendation 1. Use k3s as the short-term on-premises application runtime baseline

Continue using the k3s-based model for Linux containerized workloads where possible. Define a minimum cluster baseline, including supported OS versions, network ports, storage expectations, image pull method, certificate handling, logging, backup, upgrade, and rollback approach.

Why this solves the problem: it keeps the on-premises design close to the existing Linux-based cloud architecture and reduces duplicate engineering work while the partner operates cloud and on-premises environments in parallel.

### Recommendation 2. Use PowerShell selectively for Windows host preparation and Windows-only components

Treat PowerShell as an automation layer for Windows prerequisites, host checks, configuration, and Windows-only components rather than as the main deployment model for all workloads.

Why this solves the problem: it preserves flexibility for hospital environments that require Windows host automation while avoiding unnecessary divergence from the containerized architecture.

### Recommendation 3. Evaluate Azure Arc-enabled Kubernetes and Azure Arc-enabled servers for hybrid operations

Use Azure Arc-enabled Kubernetes as the management pattern for connected on-premises Kubernetes clusters when the k3s environment meets Azure Arc prerequisites. Use Azure Arc-enabled servers for Windows hosts where host-level visibility, script execution, and governance are needed.

Why this solves the problem: Azure Arc provides a Microsoft-aligned control plane for hybrid inventory, governance, and operations without requiring the partner to move every workload into Azure immediately.

### Recommendation 4. Establish a hospital deployment readiness checklist

Create a repeatable readiness checklist covering operating system, CPU and memory, storage, network egress, DNS, TLS, container registry access, PowerShell execution policy, logging, patching, backup, and support boundaries.

Why this solves the problem: hospital infrastructure differences can be handled through a predictable validation process before installation begins.

### Recommendation 5. Map HIPAA-related technical controls to Microsoft security and governance services

Use Microsoft Defender for Cloud, Azure Policy, Azure Monitor, and Azure Arc governance capabilities as reference points for security posture, regulatory compliance visibility, and operational monitoring.

Why this solves the problem: these services help structure the technical control discussion around measurable configuration, monitoring, and remediation activities rather than abstract compliance statements.
