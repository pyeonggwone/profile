2. HIPAA 요구사항에 적합한 온프레미스 개발 플로우

병원 클라이언트 배포를 위해 현재 3개 파일로 구성된 가이드

HIPAA 네트워크 인프라 기반층
HIPAA §164.312(e)(1)(e)(2)(ii) 전송 보안 요건을 최소 비용(~$437/월) 으로 충족하는 네트워크 설계서입니다. Azure Firewall Premium을 제거하고 S2S VPN (IKEv2 + AES-256) + NSG 최소 권한 규칙 조합으로 대체하였습니다. Hub VNet (VPN Gateway), AI Spoke VNet (AKS + Private Endpoint), Shared VNet (Key Vault + MySQL)의 3-VNet 허브-스포크 구조이며, 감사 로그는 Log Analytics 30일 + Storage Archive 6년 이중 보존으로 §164.312(b)를 충족합니다. 병원 측 온프레미스 네트워크(192.168.10.0/24)와 Azure 구간 전체를 암호화 채널로 연결하는 것이 핵심입니다.

k3s 컨테이너 방식 (Windows Server 2019/2022 전용)
병원 서버 OS가 Windows Server 2019/2022로 확정된 경우 적용하는 고도화 배포 방식입니다. 병원 서버에 k3s 단일 노드 클러스터를 구성하고 pii-masking, rabbitmq-producer, config-sync 3개 Pod를 컨테이너로 운영합니다. Azure Arc GitOps(Flux)를 통해 ISV가 중앙에서 k8s 매니페스트를 배포하므로 병원 방문 없이 업데이트가 가능합니다. Key Vault CSI Driver로 시크릿을 Pod에 주입하여 자격 증명이 디스크에 남지 않습니다. 단, 병원 IT 담당자에게 기본적인 Kubernetes 운영 지식이 요구되는 점이 제약입니다.

적용 시나리오: Windows Server 2019 / 2022 병원 서버에 k3s를 설치하고, 컨테이너 기반 클라이언트 에이전트를 Azure SaaS 백엔드와 연동
네트워크 연동 기준: hipaa-network.ver4.md — S2S VPN + NSG
HIPAA 기준: §164.312(e)(1) 전송 보안, §164.312(e)(2)(ii) 전송 중 암호화, §164.312(b) 감사 통제

병원 서버 OS 버전이 혼재하거나 불명확한 경우 적용하는 범용 배포 방식입니다. MedicalAI-Client.zip 번들 + 서명된 Install.ps1 스크립트로 Windows SCM(Service Control Manager)에 3개 Windows Service(PII Masking, RabbitMQ Producer Agent, Config Sync)를 등록합니다. k3s 방식과 달리 컨테이너 런타임이 불필요하여 OS 요구조건이 낮고 병원 IT의 기술 장벽이 최소화됩니다. 원격 업데이트는 Arc Run Command로 Install.ps1을 재실행하는 방식입니다.
