## Azure Arc-enabled Kubernetes

Azure Arc-enabled Kubernetes is used to connect and manage Kubernetes clusters that run outside Azure. For this project, it is relevant because Medical AI is considering a k3s-based on-premises model and wants to keep the on-premises architecture aligned with the current Linux-based cloud architecture. The needed function is hybrid cluster onboarding, configuration consistency, inventory, and governance review for the hospital deployment model.

| Area | Meeting-specific context | Decision item |
|---|---|---|
| Runtime baseline | k3s is the preferred short-term on-premises deployment strategy | Confirm whether the target k3s environment meets Azure Arc-enabled Kubernetes prerequisites |
| Architecture consistency | The cloud service is Linux-based and the partner wants to minimize architecture changes | Keep containerized workloads on the k3s path where possible |
| Operational ownership | Cloud and on-premises environments need to run in parallel | Define who manages cluster lifecycle, upgrades, connectivity, and support boundaries |
| Hospital constraints | US or international hospitals may have different infrastructure policies | Validate network egress, DNS, certificates, proxy, and runtime restrictions before installation |

### 1. Validate cluster prerequisites

1. Azure portal > Azure Arc > Kubernetes clusters > Add > Connect an existing Kubernetes cluster.
   - Check: supported Kubernetes distribution and version
   - Check: outbound connectivity requirements
   - Check: DNS resolution and proxy requirements
   - Check: cluster-admin access for onboarding
2. Azure portal > Azure Arc > Kubernetes clusters > Add > Prerequisites.
   - Check: Azure CLI and connectedk8s extension availability
   - Check: subscription and resource group target
   - Check: region for Azure Arc metadata

### 2. Connect the on-premises k3s cluster

1. Azure portal > Azure Arc > Kubernetes clusters > Add > Generate script.
   - Input: subscription
   - Input: resource group
   - Input: cluster name
   - Input: region
2. On-premises admin workstation > Terminal > Generated Azure CLI command.
   - Check: kubeconfig points to the target k3s cluster
   - Check: Azure Arc agents are created in the cluster
   - Check: cluster status becomes Connected in Azure portal

### 3. Configure GitOps-based deployment consistency

1. Azure portal > Azure Arc > Kubernetes clusters > target cluster > GitOps > Create.
   - Input: configuration name
   - Input: repository URL
   - Input: branch
   - Input: path to Kubernetes manifests
   - Option: namespace scope or cluster scope
2. Azure portal > Azure Arc > Kubernetes clusters > target cluster > GitOps.
   - Check: compliance state
   - Check: last sync status
   - Check: deployment namespace and workload status

## Azure Arc-enabled servers

Azure Arc-enabled servers is used to manage Windows or Linux servers outside Azure through Azure management capabilities. For this project, it is relevant because Medical AI is considering a hybrid approach where PowerShell-based installation is selectively applied to Windows-based hosts. The needed function is host inventory, controlled script execution, and operational visibility for Windows components that cannot be fully handled through the k3s runtime.

| Area | Meeting-specific context | Decision item |
|---|---|---|
| Windows host scope | Some components may require Windows-based hosts | Confirm which components stay outside the k3s runtime |
| PowerShell usage | PowerShell is useful where container runtime introduction is difficult | Use PowerShell for prerequisites, host checks, and Windows-only components |
| Support model | Dedicated infrastructure has reduced OS inconsistency in domestic deployments | Define the minimum supported Windows Server versions and patch baseline |
| Hospital constraints | International hospitals may apply different infrastructure policies | Validate execution policy, local permissions, network egress, and audit requirements |

### 1. Connect Windows hosts

1. Azure portal > Azure Arc > Machines > Add/Create > Add a machine.
   - Input: subscription
   - Input: resource group
   - Input: region
   - Input: operating system
2. Azure portal > Azure Arc > Machines > Add/Create > Generate script.
   - Option: single server
   - Option: Windows
3. Windows host > PowerShell > Generated onboarding script.
   - Check: Azure Connected Machine agent installed
   - Check: machine status becomes Connected in Azure portal

### 2. Run controlled PowerShell operations

1. Azure portal > Azure Arc > Machines > target machine > Run command.
   - Input: command name
   - Input: PowerShell script
   - Check: command status
   - Check: command output
2. Azure portal > Azure Arc > Machines > target machine > Activity log.
   - Check: execution timestamp
   - Check: initiating identity
   - Check: result state

### 3. Review host readiness and updates

1. Azure portal > Azure Arc > Machines > target machine > Overview.
   - Check: operating system
   - Check: agent status
   - Check: resource group and tags
2. Azure portal > Azure Arc > Machines > target machine > Update management.
   - Check: assessment status
   - Check: missing updates
   - Check: maintenance window requirements

## Microsoft Defender for Cloud

Microsoft Defender for Cloud is used to assess security posture, recommendations, and regulatory compliance views across Azure and supported hybrid resources. For this project, it is relevant because Medical AI is reviewing the on-premises deployment approach in consideration of HIPAA requirements. The needed function is to use Microsoft security posture and regulatory compliance references to structure the technical control review for hospital deployments.

| Area | Meeting-specific context | Decision item |
|---|---|---|
| HIPAA consideration | The on-premises deployment approach is being reviewed with HIPAA requirements in mind | Use compliance references as guidance for technical control mapping |
| Hybrid resources | k3s clusters and Windows hosts may exist outside Azure | Confirm which resources can be represented through Azure Arc and assessed in Defender for Cloud |
| Evidence needs | Hospital reviews may require operational evidence | Identify which recommendations, logs, and assessments can be exported or documented |
| Scope boundary | Microsoft guidance does not replace the partner's compliance responsibility | Separate technical implementation evidence from legal compliance attestation |

### 1. Open regulatory compliance view

1. Azure portal > Microsoft Defender for Cloud > Regulatory compliance.
   - Check: available standards
   - Check: healthcare-related standards applicable to the subscription
   - Check: failed controls and recommendations
2. Azure portal > Microsoft Defender for Cloud > Environment settings.
   - Check: target subscription
   - Check: Defender plans
   - Check: Azure Arc-connected resource coverage

### 2. Review security recommendations

1. Azure portal > Microsoft Defender for Cloud > Recommendations.
   - Filter: resource type
   - Filter: severity
   - Filter: regulatory compliance control
2. Azure portal > Microsoft Defender for Cloud > Recommendations > target recommendation.
   - Check: affected resources
   - Check: remediation steps
   - Check: exemption option

### 3. Export assessment evidence

1. Azure portal > Microsoft Defender for Cloud > Regulatory compliance > Download report.
   - Output: compliance report
   - Check: selected standard
   - Check: assessment date
2. Azure portal > Microsoft Defender for Cloud > Workflow automation.
   - Input: trigger condition
   - Input: target Logic App
   - Check: notification or evidence collection flow
