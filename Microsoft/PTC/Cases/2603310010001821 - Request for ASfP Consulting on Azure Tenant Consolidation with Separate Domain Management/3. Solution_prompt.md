# Question-by-question Solution Writer

현재 케이스 폴더 기준으로 아래 파일을 지정된 순서대로 읽고, 한국어 고객 회신용 `3. Solution.md`를 작성하라. `.md`는 길게 쓰라는 의미가 아니라, 고객 질문에 대한 답을 명확한 구조로 쓰라는 의미다.

## MUST READ

1. `reference/case_statement.md`
   - 고객 문의 원문을 확인한다.
   - 고객이 실제로 묻는 질문을 빠짐없이 추출한다.
   - 고객 원문 질문의 범위를 벗어난 답변은 작성하지 않는다.

2. `1. note.md`
   - `scoping`, `requirement_evaluation`, `research_notes.facts`, `research_notes.follow_up_questions`, `research_notes.references`를 확인한다.
   - 답변의 기술 reference는 `research_notes.facts`, `research_notes.references`, 공식 문서로 확인 가능한 내용 중심으로 작성한다.

3. `reference/reference_link.md`
   - 공식 문서 URL 출처를 확인한다.
   - 답변에 직접 사용한 제품 동작, 제약, 설계 옵션의 공식 문서 링크만 포함한다.
   - 관련 없는 일반 링크, 포털 URL, 전체 링크 목록은 나열하지 않는다.

4. `2. Portal Step Guide.json`
   - 포털에서 무엇을 확인해야 하는지만 `sections`, `tasks`, `steps`, `fields` 구조를 기준으로 파악한다.
   - 고객 답변에 필요한 검증 포인트만 요약한다.
   - Portal URL, Docs URL, 전체 numbered steps는 복사하지 않는다.

## SOURCE PRIORITY

서로 충돌하는 내용이 있으면 아래 순서를 따른다.

1. 현재 사용자 요청
2. `reference/case_statement.md`
3. `1. note.md`
4. Microsoft 공식 문서
5. `reference/reference_link.md`
6. `2. Portal Step Guide.json`

## OUTPUT RULES

`3. Solution.md`는 고객에게 전달할 본문만 포함한다.

내부 작업 파일명은 고객 본문에 쓰지 않는다. 고객 본문에는 `reference/case_statement.md`, `1. note.md`, `reference/reference_link.md`, `2. Portal Step Guide.json` 같은 파일명을 쓰지 않는다.

반드시 아래 방식으로 작성한다.

1. 고객 질문 추출
   - 고객 원문에서 질문을 항목별로 추출한다.
   - 원문 인용은 짧게만 사용한다.

2. 질문별 직접 답변
   - 각 질문마다 `답변`, `권장 설계`, `제약/주의`, `reference`를 쓴다.
   - 가능 여부를 먼저 쓴다: 가능 / 가능하나 제약 있음 / 권장하지 않음 / 별도 검토 필요.
   - 답변을 추가 정보 요청으로 대체하지 않는다.
   - 추가 확인이 필요한 경우에도 먼저 현재 기준의 답을 쓴 뒤, 마지막에 설계 확정에 필요한 값으로 분리한다.

3. Best practice
   - 고객 질문과 직접 연결되는 best practice만 작성한다.
   - 무관한 아키텍처 링크나 이미지 링크로 분량을 늘리지 않는다.

4. reference
   - 답변에 사용한 공식 문서 링크만 표로 정리한다.
   - 공식 문서 원문 인용은 답변 판단을 뒷받침하는 reference text로 짧게 사용한다.

5. 다음 단계
   - 고객이 제공해야 하는 값은 마지막에 별도 표로 정리한다.
   - 이 표는 책임 전가가 아니라 설계 확정을 위한 입력값이어야 한다.

## DO NOT

- 포괄적인 일반론 이메일로 바꾸지 않는다.
- 고객 질문에 없는 내용을 장황하게 확장하지 않는다.
- “추가 확인 필요”만 반복하며 답변을 회피하지 않는다.
- 내부 파일명, 내부 작업 산출물명, Portal URL, 전체 절차를 고객 본문에 노출하지 않는다.
- 공식 문서로 확인되지 않은 제품 동작을 확정 표현으로 쓰지 않는다.
- `.md`라는 이유로 무작정 길게 쓰지 않는다.
