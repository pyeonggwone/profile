# Solution-Step Merger

현재 케이스 폴더 기준으로 `2. Portal Step Guide.md`와 `3. Solution.md`를 읽고, 두 문서를 병합한 고객 회신/실행 검토용 `4. Final.md`를 작성하라.

이 단계의 목적은 새 답변을 다시 쓰는 것이 아니라, `3. Solution.md`의 질문별 답변 구조를 유지하면서 각 답변 항목에 `2. Portal Step Guide.md`의 실제 구현 방법을 연결하는 것이다.

## MUST READ

1. `3. Solution.md`
   - 고객 질문별 답변을 기준 문서로 사용한다.
   - 질문, 직접 답변, 권장 설계, 제약/주의, reference를 크게 수정하지 않는다.
   - 답변 구조를 포괄적인 일반론으로 바꾸지 않는다.

2. `2. Portal Step Guide.md`
   - 각 답변 항목과 연결되는 portal 구현 단계를 확인한다.
   - 실제 구현에 필요한 portal menu path, object type, sample value, validation point를 추출한다.
   - 단계별 구현 방법은 필요한 범위에서만 정리한다.

## SOURCE PRIORITY

서로 충돌하는 내용이 있으면 아래 순서를 따른다.

1. 현재 사용자 요청
2. `3. Solution.md`
3. `2. Portal Step Guide.md`

## MERGE RULES

- `3. Solution.md`의 질문별 답변 순서를 유지한다.
- 각 질문별 섹션에 `구현 방법` 또는 `Portal 검증 단계`를 추가한다.
- `2. Portal Step Guide.md`의 단계는 solution 항목과 맞는 것만 연결한다.
- solution에 없는 새 제품/기능/아키텍처 판단을 추가하지 않는다.
- step guide에 있는 sample tenant ID, domain, user, group, role, policy 이름은 예시값으로만 표시한다.
- production 변경 가능성이 있는 단계는 “적용”이 아니라 “pilot / report-only / 검증”으로 표현한다.
- 단순 병합이 목적이므로 3번 답변 문장을 대폭 재작성하지 않는다.

## OUTPUT STRUCTURE

`4. Final.md`는 고객이 질문별 답변과 실제 구현 검토 방법을 한 번에 볼 수 있게 작성한다.

1. 첫 블록
   - 단일 tenant + 회사별 domain 분리 운영 가능 여부를 짧게 요약한다.
   - 이 문서는 답변과 구현 검증 단계를 연결한 문서임을 밝힌다.

2. 질문별 구현 매핑
   - 각 고객 질문별로 아래 구조를 반복한다.
   - `답변`: 3번 solution의 직접 답변 유지
   - `구현 방법`: 2번 step guide에서 연결되는 portal 작업 정리
   - `검증 포인트`: 실제 설정 후 확인해야 할 값
   - `주의`: production 적용 전 확인해야 할 제약
   - `reference`: 3번 solution의 reference 유지

3. 전체 pilot 순서
   - tenant baseline 확인
   - custom domain 검증
   - user / group 표준 구성
   - Administrative unit 및 delegated role 구성
   - Conditional Access report-only 검증
   - Microsoft 365 domain 검토
   - 기존 tenant 공존 검토

4. 입력값 정리
   - 2번 step guide의 설정값 입력값을 그대로 실행값처럼 쓰지 않는다.
   - sample value / customer value column으로 나누어 고객이 실제 값을 채울 수 있게 한다.

## DO NOT

- 기존 `2. Portal Step Guide.md` 또는 `3. Solution.md`를 삭제하거나 대폭 수정하지 않는다.
- 병합 과정에서 solution의 답변 결론을 바꾸지 않는다.
- step guide에 없는 구현 단계를 새로 만들어내지 않는다.
- Portal URL, Docs URL을 본문에 그대로 나열하지 않는다.
- 내부 파일명 중심으로 설명하지 않는다. 고객 본문에서는 “답변”, “구현 방법”, “검증 포인트” 같은 업무 용어를 사용한다.
