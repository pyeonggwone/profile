## 1. 단일 tenant 통합 가능 여부

**답변: 가능합니다.** 여러 계열사의 identity를 하나의 Microsoft Entra tenant로 통합하는 target model은 검토할 수 있습니다. 이 경우 단일 tenant가 identity, authentication, access policy, directory role의 공통 운영 경계가 됩니다.

권장 설계는 그룹 공통 Microsoft Entra tenant를 target tenant로 정의하고, 계열사 구분은 company attribute, group, Administrative unit, naming standard로 처리하는 방식입니다. 관리자 모델은 중앙 Global Administrator / Security Administrator와 회사별 scoped administrator를 분리하고, 정책 모델은 공통 security baseline과 회사별 Conditional Access scope를 조합하는 형태가 적절합니다.

**구현 방법**

- Microsoft Entra admin center > Identity > Overview
- Target tenant baseline
	- Tenant name: `Korea Enterprise Group Unified Tenant`
	- Tenant ID: `8f3c2a9e-5d41-4c7a-9b28-6e1f0d3a72c4`
	- Primary domain: `kegroup.co.kr`
- Existing affiliated tenant baseline
	- Tenant name: `Korea Enterprise Manufacturing`
	- Tenant ID: `4c9a7d20-6f82-45e1-9c34-0b7e6d219f5a`
	- Primary domain: `keg-manufacturing.co.kr`
- New Azure adoption entity baseline
	- Entity name: `Korea Enterprise Logistics`
	- Planned domain: `keg-logistics.co.kr`
	- Identity source: `Cloud-only users`

**주의할 점:** 단일 tenant 통합은 운영 표준화에는 유리하지만, 계열사별 독립 tenant boundary를 유지하는 것과는 다릅니다. Microsoft 문서에서도 tenant는 identity and access management의 운영 범위로 설명됩니다. 계열사별 규제, audit ownership, data residency, 보안 책임 분리가 tenant 단위로 요구된다면 단일 tenant 통합 대신 multi-tenant 유지 또는 phased coexistence도 선택지로 남겨야 합니다.

Reference: [Multitenant organization capabilities in Microsoft Entra ID](https://learn.microsoft.com/en-us/entra/identity/multi-tenant-organizations/overview)

## 2. 회사별 email 및 identity domain 분리 운영

**답변: 가능합니다.** 단일 Microsoft Entra tenant에 여러 custom domain을 추가하고, 사용자별 UPN suffix 또는 email domain으로 각 회사 domain을 사용할 수 있습니다.

Microsoft Learn은 custom domain에 대해 “you can add your organization's DNS name as a custom domain name and set it as primary”라고 설명합니다. 이 구조를 사용하면 하나의 tenant 안에서도 `keg-manufacturing.co.kr`, `keg-logistics.co.kr`, `keg-retail.co.kr` 같은 회사별 identity domain을 운영할 수 있습니다.

**구현 방법**

- Microsoft Entra admin center > Identity > Settings > Domain names
- Add custom domain
	- Custom domain name: `keg-manufacturing.co.kr`
- Add custom domain
	- Custom domain name: `keg-logistics.co.kr`
- Add custom domain
	- Custom domain name: `keg-retail.co.kr`
- `keg-manufacturing.co.kr` > DNS record values
	- Record type: `TXT`
	- Alias or host name: `@`
	- Destination or points to address: `MS=ms47283916`
- Verify

- Microsoft 365 admin center > Settings > Domains
- Add domain
	- Domain name: `keg-manufacturing.co.kr`
- Add domain
	- Domain name: `keg-logistics.co.kr`
- `keg-manufacturing.co.kr` > DNS records
	- MX: `keg-manufacturing-co-kr.mail.protection.outlook.com`
	- TXT: `v=spf1 include:spf.protection.outlook.com -all`
	- CNAME: `autodiscover.outlook.com`

**중요 제약:** 동일한 custom domain은 동시에 여러 tenant에서 verify할 수 없습니다. Microsoft Learn은 “A domain name can only be verified in one directory”라고 명시합니다. 따라서 기존 계열사 tenant에서 사용 중인 domain을 target tenant에 등록하려면 domain 제거 / 이전 순서, 사용자 로그인 영향, Exchange mail flow 영향, 관련 workload 영향을 먼저 설계해야 합니다.

Reference: [Add your custom domain name to your tenant](https://learn.microsoft.com/en-us/entra/fundamentals/add-custom-domain)

## 3. unified tenant에서 여러 회사 domain 관리 방식

**답변: company domain은 custom domain으로 관리하고, 운영 경계는 identity object와 governance object로 나누어 관리하는 방식이 적절합니다.**

domain ownership은 회사별 custom domain의 DNS TXT 또는 MX record로 확인하고, user identity는 회사별 UPN suffix, company attribute, department attribute로 표준화합니다. group은 회사별 user group, admin group, application access group으로 분리하고, role delegation은 Administrative unit과 scoped role assignment를 활용합니다. Conditional Access 정책은 회사별 group 또는 admin group 기준으로 scope를 지정합니다.

**구현 방법**

- Microsoft Entra admin center > Identity > Users > All users
- New user > Create new user
	- User principal name: `minjun.kim@keg-manufacturing.co.kr`
	- Display name: `Minjun Kim`
	- Company name: `Korea Enterprise Manufacturing`
	- Department: `Plant Operations`
- New user > Create new user
	- User principal name: `seoyeon.park@keg-logistics.co.kr`
	- Display name: `Seoyeon Park`
	- Company name: `Korea Enterprise Logistics`
	- Department: `Logistics Operations`
- Microsoft Entra admin center > Identity > Groups > All groups
- New group
	- Group type: `Security`
	- Group name: `KEG Manufacturing Users`
	- Membership type: `Assigned`
	- Members: `minjun.kim@keg-manufacturing.co.kr`
- New group
	- Group type: `Security`
	- Group name: `KEG Logistics Users`
	- Membership type: `Assigned`
	- Members: `seoyeon.park@keg-logistics.co.kr`

이 방식은 회사별 운영을 “구분”하는 모델이지, 회사별 tenant를 “분리”하는 모델은 아닙니다. 완전한 보안 경계가 필요하면 단일 tenant 통합 대신 multi-tenant coexistence도 같이 검토해야 합니다.

## 4. identity, authentication, governance 모델

**답변: 중앙 통제 + 회사별 위임 운영 모델을 권장합니다.** 단일 tenant 사용자 객체와 회사별 UPN suffix / attribute를 조합하면 사용자 표준화와 domain 분리 운영을 동시에 달성할 수 있습니다. Authentication은 공통 baseline과 회사별 Conditional Access 조건을 조합하고, Governance는 Administrative unit, scoped role, security group으로 구성하는 방식이 현실적입니다.

Administrative unit에 대해서 Microsoft Learn은 “Administrative units restrict permissions in a role to any portion of your organization that you define”라고 설명합니다. 따라서 회사별 User Administrator 또는 Group Administrator 위임에는 적합합니다. 다만 같은 문서에서 “Administrative units apply scope only to management permissions”라고 설명하므로, Administrative unit을 tenant isolation처럼 해석하면 안 됩니다.

**구현 방법**

- Microsoft Entra admin center > Identity > Roles & admins > Admin units
- Add
	- Name: `AU-KEG-Manufacturing`
	- Description: `Administrative unit for Korea Enterprise Manufacturing identities`
- Add
	- Name: `AU-KEG-Logistics`
	- Description: `Administrative unit for Korea Enterprise Logistics identities`
- `AU-KEG-Manufacturing` > Users > Add members
	- Members: `minjun.kim@keg-manufacturing.co.kr`
- `AU-KEG-Manufacturing` > Groups > Add members
	- Members: `KEG Manufacturing Users`
- `AU-KEG-Manufacturing` > Roles and administrators
- Add assignments
	- Role: `User Administrator`
	- Select member: `identity-admin-mfg@kegroup.co.kr`
- Microsoft Entra admin center > Identity > Roles & admins > Roles & admins
- `Global Administrator` > Assignments
	- Member: `tenant-platform-admin@kegroup.co.kr`
- `Security Administrator` > Assignments
	- Member: `security-admin@kegroup.co.kr`
- Microsoft Entra admin center > Protection > Conditional Access > Policies
- New policy
	- Name: `CA-KEG-Manufacturing-Admins-Require-MFA`
	- Users: `KEG Manufacturing Admins`
	- Target resources: `All cloud apps`
	- Grant: `Require multifactor authentication`
	- Enable policy: `Report-only`

Reference: [Administrative units in Microsoft Entra ID](https://learn.microsoft.com/en-us/entra/identity/role-based-access-control/administrative-units), [What is Conditional Access?](https://learn.microsoft.com/en-us/entra/identity/conditional-access/overview)

## 5. 제한 사항, 위험, 운영 고려사항

**답변: 가장 큰 위험은 domain 이전, tenant-level policy 영향, 완전한 관리자 격리 한계, 기존 tenant 의존성입니다.**

Custom domain은 한 directory에서만 verify 가능하므로, 기존 tenant에 남아 있는 domain은 target tenant에 바로 추가할 수 없습니다. 이 항목은 Microsoft Entra admin center > Identity > Settings > Domain names에서 회사별 domain 상태를 확인하고, Microsoft 365가 포함된 경우 Microsoft 365 admin center > Settings > Domains의 email domain 상태와 함께 봐야 합니다.

Tenant-level policy는 Conditional Access > Policies에서 대상 users / groups / cloud apps가 과도하게 넓지 않은지 살펴야 합니다. 중앙 정책이 전체 tenant에 영향을 줄 수 있으므로, 회사별 group을 target으로 두고 초기에는 `Report-only`로 운영하는 접근이 안전합니다.

관리자 분리 한계는 Administrative unit과 Roles & admins를 함께 봐야 합니다. 회사별 admin이 scoped role인지, tenant-wide role인지가 핵심입니다. Administrative unit은 관리 권한 scope를 제한하지만 데이터, 감사, 정책, 보안 경계를 tenant처럼 완전히 분리하지는 않습니다.

기존 tenant 의존성은 tenant overview, enterprise applications, subscriptions, Microsoft 365 domains, Conditional Access policies에서 함께 정리해야 합니다. 기존 app registration, enterprise application, SSO, mail flow, subscription ownership이 남아 있다면 tenant consolidation은 identity 변경만으로 끝나지 않습니다.

Cross-tenant synchronization은 여러 tenant 간 B2B collaboration user 생성 / 업데이트 / 삭제를 자동화하는 기능입니다. 다만 Microsoft Learn FAQ는 “Cross-tenant synchronization isn't a migration tool”이라고 명시합니다. 따라서 기존 tenant가 있는 계열사는 cross-tenant synchronization을 tenant migration 대체 수단이 아니라, 단계적 공존 또는 협업 개선 수단으로 보는 것이 맞습니다.

운영 점검 항목:

- Microsoft Entra admin center > Identity > External Identities > Cross-tenant access settings
- Organizational settings
- Inbound / outbound collaboration scope

Reference: [What is cross-tenant synchronization?](https://learn.microsoft.com/en-us/entra/identity/multi-tenant-organizations/cross-tenant-synchronization-overview)

## 6. Microsoft best practice 관점의 enterprise group 구조

**답변: Microsoft Entra tenant 구조와 Azure subscription / management group 구조를 분리해서 설계하는 것이 적절합니다.**

단일 tenant 여부는 identity boundary 결정이고, enterprise group의 Azure 운영 구조는 management group, subscription, landing zone, Azure Policy, RBAC 설계로 결정됩니다. 두 결정을 하나로 묶으면 안 됩니다.

Microsoft Entra tenant는 그룹 공통 identity plane으로 사용할지 결정하는 영역입니다. 회사별 identity boundary는 UPN suffix, group, Administrative unit으로 표현합니다. Authentication baseline은 공통 baseline과 회사별 exception을 조합하고, email domain boundary는 Microsoft 365 admin center의 Domains와 DNS records 설계로 연결됩니다. 기존 tenant 공존이 필요하면 Cross-tenant access settings에서 collaboration scope를 별도로 설계합니다.

Azure landing zone 문서는 reference architecture를 “opinionated target architecture”라고 설명하며, design areas에 Microsoft Entra tenant, identity and access management, management group and subscription organization, security, governance가 포함됩니다. 따라서 고객 시나리오에서는 tenant 통합 여부와 함께 Azure landing zone 기준의 subscription / management group 설계를 병행하는 것이 Microsoft best practice에 더 가깝습니다.

Reference: [What is an Azure landing zone?](https://learn.microsoft.com/en-us/azure/cloud-adoption-framework/ready/landing-zone/)

## 7. 한국어 기반 심화 논의

**답변: 본 건은 Microsoft Entra ID tenant consolidation, multi-domain identity design, governance model 중심의 한국어 설계 논의로 진행하는 것이 적절합니다.** 후속 논의에서는 아래 산출물을 기준으로 한국어로 architecture review를 진행하면 됩니다.

- Target tenant model: full consolidation / phased coexistence / long-term multi-tenant 중 선택
- Domain transition: 회사별 custom domain 이전 순서 및 영향도
- Delegated administration: 회사별 Administrative unit, role assignment, admin group 설계
- Authentication policy: 회사별 Conditional Access baseline 및 exception 설계
- Azure organization: management group, subscription, landing zone 정렬

## 권장 pilot 흐름

1. Target tenant와 기존 tenant의 tenant name, tenant ID, primary domain, verified domain을 정리합니다.
2. 회사별 custom domain을 target tenant에 추가하고 DNS ownership record를 준비합니다.
3. 회사별 test user를 만들고 UPN suffix, company attribute, department, usage location을 표준화합니다.
4. 회사별 user group, admin group을 만들고 application access와 policy target의 기준으로 사용합니다.
5. 회사별 Administrative unit을 만들고 users / groups를 배치한 뒤 scoped role을 할당합니다.
6. Conditional Access 정책은 회사별 group 대상으로 만들고 초기에는 `Report-only`로 운영합니다.
7. Microsoft 365 email domain과 기존 tenant coexistence 항목은 mail flow와 collaboration 영향이 보이는 범위에서 별도 일정으로 다룹니다.
