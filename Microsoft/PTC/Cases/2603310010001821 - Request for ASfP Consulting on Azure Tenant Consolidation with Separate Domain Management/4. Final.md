아래 내용은 여러 계열사의 Azure / Microsoft Entra tenant 통합 및 회사별 email / identity domain 분리 운영에 대한 질문별 답변과, 해당 답변을 실제로 검증하기 위한 portal 구현 단계를 연결한 내용입니다.

> 결론: 단일 Microsoft Entra tenant 안에서 여러 custom domain을 등록하고 회사별 UPN / email domain을 운영하는 구성은 가능합니다. 다만 회사별로 완전히 독립된 tenant 수준의 격리를 제공하는 구조는 아니므로, domain 이전, policy scope, 관리자 위임, 기존 tenant 의존성은 pilot 단계에서 검증해야 합니다.

## 1. 여러 계열사의 Azure tenants를 단일 tenant로 통합할 수 있는가?

**답변:** 가능합니다. 통합 후 identity / access management의 기본 경계는 단일 Microsoft Entra tenant가 됩니다.

### 구현 방법

| 단계 | Portal 작업 | 예시 값 |
|---|---|---|
| Target tenant baseline 확인 | Microsoft Entra admin center > Identity > Overview에서 통합 기준 tenant 정보를 확인 | Tenant name: `Korea Enterprise Group Unified Tenant` |
| Target tenant ID 확인 | tenant overview에서 Tenant ID 확인 | `8f3c2a9e-5d41-4c7a-9b28-6e1f0d3a72c4` |
| Primary domain 확인 | tenant overview에서 Primary domain 확인 | `kegroup.co.kr` |
| 기존 계열사 tenant baseline 확인 | 기존 tenant의 Tenant name, Tenant ID, Primary domain 확인 | `Korea Enterprise Manufacturing`, `keg-manufacturing.co.kr` |
| 신규 도입 계열사 baseline 정리 | 신규 Azure 도입 회사의 planned domain과 identity source 정리 | `keg-logistics.co.kr`, `Cloud-only users` |

### 검증 포인트

| 확인 항목 | 확인 내용 |
|---|---|
| 기준 tenant | 실제 통합 기준 tenant가 하나로 확정되었는지 확인 |
| 기존 tenant | 기존 계열사 tenant의 domain, subscription, identity source가 식별되었는지 확인 |
| 신규 계열사 | 신규 도입 회사의 planned domain과 user source가 확인되었는지 확인 |

### 주의

단일 tenant 통합은 identity 운영 경계를 하나로 만드는 결정입니다. 기존 tenant의 subscription, app registration, enterprise application, mail flow, Conditional Access dependency가 확인되기 전에는 production 이전을 진행하지 않는 것이 적절합니다.

Reference: [Multitenant organization capabilities in Microsoft Entra ID](https://learn.microsoft.com/en-us/entra/identity/multi-tenant-organizations/overview)

## 2. 단일 tenant 안에서 회사별 email 및 identity domain을 분리 운영할 수 있는가?

**답변:** 가능합니다. 여러 custom domain을 하나의 tenant에 추가하고 사용자 UPN / email suffix로 회사별 domain을 사용할 수 있습니다.

### 구현 방법

| 단계 | Portal 작업 | 예시 값 |
|---|---|---|
| Identity domain 추가 | Microsoft Entra admin center > Identity > Settings > Domain names에서 Add custom domain | `keg-manufacturing.co.kr`, `keg-logistics.co.kr`, `keg-retail.co.kr` |
| Domain ownership 검증 | 각 domain의 DNS record values를 registrar에 등록한 뒤 Verify 수행 | TXT record, Alias: `@`, Destination: `MS=...` |
| Email domain 추가 | Microsoft 365 admin center > Settings > Domains에서 회사별 domain 추가 | `keg-manufacturing.co.kr`, `keg-logistics.co.kr` |
| Email service DNS 구성 | Microsoft 365 domain DNS records에서 MX, SPF TXT, Autodiscover CNAME 구성 검토 | MX, TXT, CNAME |

### 검증 포인트

| 확인 항목 | 확인 내용 |
|---|---|
| Domain 상태 | 회사별 custom domain이 Verified 상태인지 확인 |
| Domain 중복 | 동일 domain이 기존 tenant에 남아 verify되어 있지 않은지 확인 |
| Email 영향 | Microsoft 365 / Exchange Online 포함 시 MX, SPF, Autodiscover 영향 확인 |

### 주의

동일한 custom domain은 동시에 여러 tenant에서 verify할 수 없습니다. 기존 계열사 tenant에서 이미 사용 중인 domain은 target tenant에 추가하기 전에 이전 순서와 사용자 로그인 / mail flow 영향을 검토해야 합니다.

Reference: [Add your custom domain name to your tenant](https://learn.microsoft.com/en-us/entra/fundamentals/add-custom-domain)

## 3. 여러 회사 domain을 unified tenant에서 어떻게 관리하는가?

**답변:** 회사별 domain은 custom domain으로 관리하고, 운영 경계는 user attribute, group, Administrative unit, role assignment로 구분하는 방식이 적절합니다.

### 구현 방법

| 단계 | Portal 작업 | 예시 값 |
|---|---|---|
| 회사별 사용자 생성 | Microsoft Entra admin center > Identity > Users > All users에서 New user 생성 | `minjun.kim@keg-manufacturing.co.kr`, `seoyeon.park@keg-logistics.co.kr` |
| 사용자 속성 표준화 | user properties에서 Company name, Department, Usage location 설정 | `Korea Enterprise Manufacturing`, `Plant Operations`, `Korea` |
| 회사별 group 생성 | Identity > Groups > All groups에서 Security group 생성 | `KEG Manufacturing Users`, `KEG Logistics Users` |
| 관리자 group 생성 | 회사별 admin group을 별도로 생성 | `KEG Manufacturing Admins` |

### 검증 포인트

| 확인 항목 | 확인 내용 |
|---|---|
| UPN suffix | 사용자 UPN이 회사별 domain suffix를 사용하는지 확인 |
| Company attribute | 사용자 속성 기준으로 회사별 필터링이 가능한지 확인 |
| Group membership | 사용자 / 관리자 group이 회사별로 분리되어 있는지 확인 |

### 주의

이 구성은 회사별 운영 단위를 구분하는 방식입니다. 회사별 tenant를 별도로 유지하는 수준의 보안 격리를 제공하는 것은 아닙니다.

## 4. multi-entity 환경에 적합한 identity, authentication, governance 모델은 무엇인가?

**답변:** 중앙 tenant + 회사별 Administrative unit / security group / scoped role / Conditional Access scope 조합이 현실적인 기본 모델입니다.

### 구현 방법

| 영역 | Portal 작업 | 예시 값 |
|---|---|---|
| Administrative unit 생성 | Identity > Roles & admins > Admin units에서 회사별 administrative unit 생성 | `AU-KEG-Manufacturing`, `AU-KEG-Logistics` |
| Administrative unit member 구성 | 각 administrative unit에 회사별 users / groups 추가 | `KEG Manufacturing Users`, `KEG Logistics Users` |
| 회사별 delegated admin 지정 | Administrative unit > Roles and administrators에서 role assignment 추가 | Role: `User Administrator`, Member: `identity-admin-mfg@kegroup.co.kr` |
| 중앙 tenant admin 지정 | Roles & admins에서 중앙 운영 role assignment 구성 | `Global Administrator`, `Security Administrator`, `Conditional Access Administrator` |
| 관리자 인증 정책 | Conditional Access > Policies에서 관리자 group 대상 MFA 정책 생성 | `CA-KEG-Manufacturing-Admins-Require-MFA` |
| 일반 사용자 인증 정책 | Conditional Access > Policies에서 사용자 group 대상 compliant device 정책 생성 | `CA-KEG-Logistics-Users-Require-Compliant-Device` |

### 검증 포인트

| 확인 항목 | 확인 내용 |
|---|---|
| Administrative unit scope | 회사별 위임 관리자가 지정된 사용자 / group 범위만 관리하는지 확인 |
| Role assignment | 중앙 role과 회사별 scoped role이 구분되어 있는지 확인 |
| Conditional Access mode | 초기 정책이 `Report-only`로 설정되어 영향도 검증이 가능한지 확인 |

### 주의

Administrative unit은 관리 권한 scope를 제한하는 기능입니다. 데이터, 감사, 정책, 보안 경계를 tenant처럼 완전히 분리하는 기능으로 해석하면 안 됩니다. Conditional Access도 production 적용 전 report-only로 영향도를 확인하는 것이 적절합니다.

Reference: [Administrative units in Microsoft Entra ID](https://learn.microsoft.com/en-us/entra/identity/role-based-access-control/administrative-units), [What is Conditional Access?](https://learn.microsoft.com/en-us/entra/identity/conditional-access/overview)

## 5. 주요 제한, 위험, 운영 고려사항은 무엇인가?

**답변:** custom domain 중복 검증 불가, tenant-level policy 영향, 완전한 관리자 격리 한계, 기존 tenant 의존성이 핵심 위험입니다.

### 구현 및 검증 방법

| 리스크 | Portal 검증 위치 | 확인할 내용 |
|---|---|---|
| Custom domain 중복 | Domain names | 기존 tenant에 동일 domain이 verify되어 있는지 확인 |
| Tenant-level policy 영향 | Conditional Access > Policies | policy 대상 users / groups / cloud apps가 과도하게 넓지 않은지 확인 |
| 관리자 분리 한계 | Admin units, Roles & admins | 회사별 admin이 scoped role인지 tenant-wide role인지 확인 |
| 기존 tenant 의존성 | Tenant overview, Enterprise applications, Subscriptions, Microsoft 365 Domains | subscription, app, mail flow, domain dependency 식별 |
| 단계적 공존 필요 | External Identities > Cross-tenant access settings | 기존 tenant와 target tenant 간 B2B collaboration scope 확인 |

### 주의

Cross-tenant synchronization은 tenant migration tool이 아닙니다. 기존 tenant와 target tenant 간 협업 또는 단계적 공존을 검증하는 용도로 사용하고, 실제 tenant-to-tenant migration은 별도 계획으로 분리해야 합니다.

Reference: [What is cross-tenant synchronization?](https://learn.microsoft.com/en-us/entra/identity/multi-tenant-organizations/cross-tenant-synchronization-overview)

## 6. Microsoft best practice에 맞는 enterprise group 구조는 무엇인가?

**답변:** Microsoft Entra tenant 구조와 Azure subscription / management group 구조를 분리해서 설계하는 것이 적절합니다.

### 구현 검토 방법

| 설계 계층 | 구현 검토 항목 | 연결되는 portal 검증 |
|---|---|---|
| Microsoft Entra tenant | 그룹 공통 identity plane으로 사용할지 결정 | Tenant overview, Domain names |
| 회사별 identity boundary | 회사별 UPN suffix, group, Administrative unit 설계 | Users, Groups, Admin units |
| Authentication baseline | 공통 baseline + 회사별 exception 설계 | Conditional Access report-only policies |
| Email domain boundary | 회사별 email domain과 DNS record 설계 | Microsoft 365 admin center > Domains |
| 기존 tenant 공존 | 기존 tenant와의 collaboration scope 설계 | Cross-tenant access settings |

Azure landing zone의 management group / subscription 구조는 Microsoft Entra tenant 통합 여부와 별도로 설계해야 합니다. 즉, identity plane은 단일화하더라도 Azure subscription ownership과 policy inheritance는 계열사 / workload / environment 기준으로 나눌 수 있습니다.

Reference: [What is an Azure landing zone?](https://learn.microsoft.com/en-us/azure/cloud-adoption-framework/ready/landing-zone/)

## 전체 pilot 순서

| 순서 | 작업 | 목적 |
|---|---|---|
| 1 | target tenant와 기존 tenant baseline 확인 | 통합 기준 tenant와 기존 의존성 식별 |

| 6 | Microsoft 365 Domains에서 email domain / DNS 검토 | email domain 운영 영향도 확인 |
| 7 | Cross-tenant access settings로 기존 tenant 공존 범위 검토 | full consolidation 전 단계적 공존 가능성 확인 |
