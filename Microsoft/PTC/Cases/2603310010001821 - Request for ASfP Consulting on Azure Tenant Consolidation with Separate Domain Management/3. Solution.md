## 1. 단일 tenant 통합 가능 여부

**답변: 가능합니다.** 여러 계열사의 identity를 하나의 Microsoft Entra tenant로 통합하는 target model은 검토할 수 있습니다. 이 경우 단일 tenant가 identity, authentication, access policy, directory role의 공통 운영 경계가 됩니다.

| 항목 | 권장 설계 |
|---|---|
| 기준 tenant | 그룹 공통 Microsoft Entra tenant를 target tenant로 정의 |
| 계열사 구분 | company attribute, group, Administrative unit, naming standard로 논리적 구분 |
| 관리자 모델 | 중앙 Global Administrator / Security Administrator와 회사별 scoped administrator 분리 |
| 정책 모델 | 공통 security baseline + 회사별 Conditional Access scope 조합 |

**주의할 점:** 단일 tenant 통합은 운영 표준화에는 유리하지만, 계열사별 독립 tenant boundary를 유지하는 것과는 다릅니다. Microsoft 문서에서도 tenant는 “identity and access management scope”라고 설명합니다. 계열사별 규제, audit ownership, data residency, 보안 책임 분리가 tenant 단위로 요구된다면 단일 tenant 통합 대신 multi-tenant 유지 또는 phased coexistence도 선택지로 남겨야 합니다.

근거: [Multitenant organization capabilities in Microsoft Entra ID](https://learn.microsoft.com/en-us/entra/identity/multi-tenant-organizations/overview)

## 2. 회사별 email 및 identity domain 분리 운영

**답변: 가능합니다.** 단일 Microsoft Entra tenant에 여러 custom domain을 추가하고, 사용자별 UPN suffix 또는 email domain으로 각 회사 domain을 사용할 수 있습니다.

Microsoft Learn은 custom domain에 대해 “you can add your organization's DNS name as a custom domain name and set it as primary”라고 설명합니다.

| 항목 | 답변 |
|---|---|
| Identity domain | 회사별 domain을 Microsoft Entra custom domain으로 추가 가능 |
| User sign-in name | 사용자 UPN suffix를 회사별 domain으로 구성 가능 |
| Email domain | Microsoft 365 / Exchange Online 범위가 포함될 경우 회사별 accepted domain 및 DNS record 검토 필요 |
| 운영 표준 | `company`, `department`, group naming, lifecycle rule로 회사별 사용자 구분 |

**중요 제약:** 동일한 custom domain은 동시에 여러 tenant에서 verify할 수 없습니다. Microsoft Learn은 “A domain name can only be verified in one directory”라고 명시합니다. 따라서 기존 계열사 tenant에서 사용 중인 domain을 target tenant에 등록하려면 domain 제거 / 이전 순서, 사용자 로그인 영향, Exchange mail flow 영향, 관련 workload 영향을 먼저 설계해야 합니다.

근거: [Add your custom domain name to your tenant](https://learn.microsoft.com/en-us/entra/fundamentals/add-custom-domain)

## 3. unified tenant에서 여러 회사 domain 관리 방식

**답변: company domain은 custom domain으로 관리하고, 운영 경계는 identity object와 governance object로 나누어 관리하는 방식이 적절합니다.**

| 관리 대상 | 설계 방식 |
|---|---|
| Domain ownership | 회사별 custom domain DNS TXT 또는 MX record로 ownership 검증 |
| User identity | 회사별 UPN suffix, company attribute, department attribute 표준화 |
| Group | 회사별 user group, admin group, application access group 분리 |
| Role delegation | Administrative unit에 사용자 / group을 배치하고 회사별 관리자 role assignment 검토 |
| Policy targeting | Conditional Access 정책을 회사별 group 또는 admin group 기준으로 scope 지정 |

이 방식은 회사별 운영을 “구분”하는 모델이지, 회사별 tenant를 “분리”하는 모델은 아닙니다. 완전한 보안 경계가 필요하면 단일 tenant 통합 대신 multi-tenant coexistence도 같이 검토해야 합니다.

## 4. identity, authentication, governance 모델

**답변: 중앙 통제 + 회사별 위임 운영 모델을 권장합니다.**

| 영역 | 권장 모델 | 이유 |
|---|---|---|
| Identity | 단일 tenant 사용자 객체 + 회사별 UPN suffix / attribute | 사용자 표준화와 domain 분리 운영을 동시에 달성 가능 |
| Authentication | 공통 authentication baseline + 회사별 Conditional Access 조건 | MFA, compliant device, location, application 조건을 회사별로 조정 가능 |
| Governance | Administrative unit + scoped role + security group | 회사별 제한된 사용자 / group 운영 위임 가능 |
| Privileged access | 중앙 privileged role 최소화, 회사별 scoped admin 분리 | Global Administrator 남용 방지 및 운영 책임 구분 |
| Lifecycle | 회사별 joiner / mover / leaver 프로세스 정의 | 계열사별 인사 프로세스 차이를 반영 가능 |

Administrative unit에 대해서 Microsoft Learn은 “Administrative units restrict permissions in a role to any portion of your organization that you define”라고 설명합니다. 따라서 회사별 User Administrator 또는 Group Administrator 위임에는 적합합니다.

다만 같은 문서에서 “Administrative units apply scope only to management permissions”라고 설명하므로, Administrative unit을 tenant isolation처럼 해석하면 안 됩니다.

근거: [Administrative units in Microsoft Entra ID](https://learn.microsoft.com/en-us/entra/identity/role-based-access-control/administrative-units)

Conditional Access는 Microsoft의 “Zero Trust policy engine”으로 설명되며, 사용자, group, device, application, risk 등 signal 기반으로 access control을 적용합니다. 따라서 회사별 인증 정책은 Conditional Access policy를 바로 production에 적용하기보다 report-only pilot으로 영향도를 검증하는 것이 적절합니다.

근거: [What is Conditional Access?](https://learn.microsoft.com/en-us/entra/identity/conditional-access/overview)

## 5. 제한 사항, 위험, 운영 고려사항

**답변: 가장 큰 위험은 domain 이전, tenant-level policy 영향, 완전한 관리자 격리 한계, 기존 tenant 의존성입니다.**

| 리스크 | 정확한 답변 | 대응 방향 |
|---|---|---|
| Custom domain 중복 | 동일 domain은 한 directory에서만 verify 가능 | domain 이전 순서와 DNS / mail flow 변경 계획 필요 |
| Tenant-level policy | 일부 policy, directory role, audit, identity protection은 tenant 전체에 영향 가능 | policy scope와 예외 설계를 사전에 정의 |
| 관리자 분리 한계 | Administrative unit은 관리 권한 scope만 제한 | 완전 격리 요구가 있으면 multi-tenant 유지 검토 |
| 기존 tenant 의존성 | 기존 subscription, app registration, enterprise application, mail flow, Conditional Access dependency가 통합을 복잡하게 만들 수 있음 | tenant / subscription / app / mail inventory 선행 |
| Cross-tenant coexistence | cross-tenant synchronization은 migration tool이 아님 | 공존 / 협업용으로 사용하고 migration 계획은 별도 수립 |

Cross-tenant synchronization은 여러 tenant 간 B2B collaboration user 생성 / 업데이트 / 삭제를 자동화하는 기능입니다. 다만 Microsoft Learn FAQ는 “Cross-tenant synchronization isn't a migration tool”이라고 명시합니다. 따라서 기존 tenant가 있는 계열사는 cross-tenant synchronization을 tenant migration 대체 수단이 아니라, 단계적 공존 또는 협업 개선 수단으로 보는 것이 맞습니다.

근거: [What is cross-tenant synchronization?](https://learn.microsoft.com/en-us/entra/identity/multi-tenant-organizations/cross-tenant-synchronization-overview)

## 6. Microsoft best practice 관점의 enterprise group 구조

**답변: Microsoft Entra tenant 구조와 Azure subscription / management group 구조를 분리해서 설계하는 것이 적절합니다.**

단일 tenant 여부는 identity boundary 결정이고, enterprise group의 Azure 운영 구조는 management group, subscription, landing zone, Azure Policy, RBAC 설계로 결정됩니다. 두 결정을 하나로 묶으면 안 됩니다.

| 설계 계층 | 권장 방향 |
|---|---|
| Microsoft Entra tenant | 그룹 공통 identity plane으로 사용 여부 결정 |
| Management group | 계열사 / workload / environment 기준 정책 상속 구조 설계 |
| Subscription | 회사별 또는 workload별 ownership과 billing 책임 분리 |
| Platform landing zone | identity, connectivity, management 같은 shared service 중앙 운영 |
| Application landing zone | 계열사 또는 workload team이 운영하는 application subscription 배치 |

Azure landing zone 문서는 reference architecture를 “opinionated target architecture”라고 설명하며, design areas에 Microsoft Entra tenant, identity and access management, management group and subscription organization, security, governance가 포함됩니다. 따라서 고객 시나리오에서는 tenant 통합 여부와 함께 Azure landing zone 기준의 subscription / management group 설계를 병행하는 것이 Microsoft best practice에 더 가깝습니다.

근거: [What is an Azure landing zone?](https://learn.microsoft.com/en-us/azure/cloud-adoption-framework/ready/landing-zone/)

## 7. 한국어 기반 심화 논의

**답변: 본 건은 Microsoft Entra ID tenant consolidation, multi-domain identity design, governance model 중심의 한국어 설계 논의로 진행하는 것이 적절합니다.** 후속 논의에서는 아래 산출물을 기준으로 한국어로 architecture review를 진행하면 됩니다.

| 논의 주제 | 목표 산출물 |
|---|---|
| Target tenant model | full consolidation / phased coexistence / long-term multi-tenant 중 선택 |
| Domain transition | 회사별 custom domain 이전 순서 및 영향도 |
| Delegated administration | 회사별 Administrative unit, role assignment, admin group 설계 |
| Authentication policy | 회사별 Conditional Access baseline 및 exception 설계 |
| Azure organization | management group, subscription, landing zone 정렬 |

## 설계 확정을 위해 필요한 입력값

아래 값은 답변을 회피하기 위한 전제가 아니라, 위 답변을 실제 고객 환경에 맞는 target architecture로 확정하기 위해 필요한 입력입니다.

| 필요 값 | 사용 목적 |
|---|---|
| 계열사별 현재 Azure / Microsoft Entra tenant 수, tenant ID, primary domain, verified domain | domain 이전 가능 여부와 tenant 통합 순서 결정 |
| 계열사별 subscription 수, billing owner, management group 사용 여부 | Azure landing zone 및 enterprise group 구조 설계 |
| 계열사별 identity source: cloud-only, Active Directory, Microsoft Entra Connect, Cloud Sync, federation | 사용자 source of authority와 authentication 영향도 확인 |
| Exchange Online, Teams, SharePoint, OneDrive, Power BI 포함 여부 | email domain 및 collaboration workload 영향도 확인 |
| app registration, enterprise application, SSO, Conditional Access dependency | 기존 tenant 통합 시 application 영향도 확인 |
| 회사별 audit, compliance, data residency, privileged admin separation 요구 | 단일 tenant가 적합한지 또는 multi-tenant 유지가 필요한지 판단 |
