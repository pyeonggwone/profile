# Portal Step Guide Writer
사용자의 요구사항과 참고 문서를 읽고 포털에서 그대로 사용할 수 있는 한국어 단계형 `.md` 문서를 작성하라.
동일한 결과를 자동화나 재사용에 활용할 수 있도록 `.md`와 같은 문서 구조와 단계 내용을 담은 `.json` 파일도 함께 작성하라.
입력 파일은 다음 순서로 참고하라.
1. `reference/case_statement.md`의 `[Customer Statement]`를 1차 기준으로 사용한다.
2. `1. note.md`의 `# scoping`, `# requirement_evaluation`, `# research_notes`를 보조 기준으로 사용한다.
3. 기존 solution 문서나 reference 문서는 공식 Docs URL과 세부 포털 기능 확인에만 사용하고, 케이스 범위를 바꾸는 기준으로 사용하지 않는다.
4. `[Customer Statement]`가 아키텍처 자문, 설계 옵션, 운영 고려사항을 묻는 경우 특정 구현 기능 하나만 중심으로 작성하지 않는다.
5. 고객 문의가 tenant consolidation, multi-domain, governance, authentication, risk를 포함하면 custom domain, user/domain assignment, administrative delegation, Conditional Access, role governance, inventory 확인 기능을 우선 포함한다.
문서는 `# <문서 제목>`으로 시작한다.
`## <번호>. <서비스명 또는 포털 영역명>` 형식으로 서비스 대분류를 만든다.
각 `##` 섹션 제목 아래에는 반드시 `Portal URL: <해당 기능 포털 URL>`과 `Docs URL: <공식 문서 URL>`을 순서대로 작성한다.
같은 서비스 영역에서 같은 `Portal URL`과 같은 `Docs URL`을 사용하는 작업은 별도 `##` 섹션으로 반복하지 말고 하나의 `##` 섹션으로 합친다.
합친 `##` 섹션 안에서는 `### <기능명 또는 작업명>` 하위 제목으로 생성, 설정, 할당, 테스트, 모니터링 단계를 나눈다.
`Portal URL`은 포괄적인 홈 URL이 아니라 해당 기능 화면으로 연결되는 URL을 사용한다.
`Docs URL` 아래에는 numbered steps와 필요한 `###` 하위 제목만 작성하고 설명, 배경, 요약, 비교, 고객 안내 문장은 쓰지 않는다.
단계는 목적 기준으로 나누고, 같은 값이나 같은 화면은 한 번만 다룬다.
단계 문장은 `이동한다`, `확인한다`, `기록한다`, `확인 대상`, `확인 필요`, `PLACEHOLDER`를 쓰지 않는다.
포털 경로는 `<관리 포털명> > <메뉴> > <하위 메뉴>` 형식의 명사형으로 쓴다.
입력값은 실제 형식과 유사한 예시 데이터로 작성한다.
Tenant ID, Subscription ID, Object ID 같은 GUID 값은 반복 숫자나 반복 문자로만 쓰지 말고 실제 GUID처럼 보이는 예시 값을 사용한다.
도메인, 사용자, 그룹, 앱, 정책, 리소스 이름 예시는 Microsoft 문서의 일반 샘플 값을 기본값으로 쓰지 말고 입력 케이스의 산업, 지역, 조직 구조, 목적에 맞는 예시 값으로 작성한다.
케이스에 실제 고객값이 없으면 업무 맥락에 맞는 가상 예시를 만들고, 이후 단계에서 같은 값이 필요하면 동일한 예시 값을 그대로 재사용한다.
입력값은 `Field name: `value`` 형식으로 쓰며 값은 backtick으로 감싼다.
같은 화면의 설정 필드, 대상 목록, 결과 값은 한 숫자 단계 아래 bullet list로 묶는다.
Microsoft 제품명, 서비스명, role, 메뉴명은 원문을 유지한다.
production 변경 위험이 큰 삭제, 기본 정책 차단, 전체 사용자 대상 변경은 실행 단계로 쓰지 않는다.
CLI, PowerShell, Terraform, Bicep 명령은 작성하지 않는다.
표, 완료 확인, 주의 사항, 수행 조건, 수행 절차, 결과 정리, 목적, 사전 조건, 의사결정 표, risk register는 쓰지 않는다.
작업 목록이 명시되면 해당 작업만 포함하고, 없으면 요구사항 충족에 필요한 최소 포털 작업만 포함한다.
여러 제품 영역은 제품 또는 포털 기준으로 `##` 섹션을 분리한다.
마지막 섹션은 필요한 경우 `설정값 입력값`, `고객 입력값` 같은 값 목록 코드 블록만 사용한다.
JSON 파일명은 Markdown 결과 파일명과 같은 prefix를 사용해 `2. Portal Step Guide.json`처럼 작성한다.
JSON은 함께 생성되는 Markdown 결과물과 동일한 계층 구조를 유지한다.
최상위 object는 다음 필드를 포함한다.
- `title`: Markdown의 `#` 문서 제목
- `sections`: Markdown의 `##` 섹션 배열
각 `sections` 항목은 다음 필드를 포함한다.
- `section_no`: `##` 섹션 번호
- `title`: `##` 섹션 제목에서 번호를 제외한 값
- `portal_url`: 현재 섹션의 Portal URL
- `docs_url`: 현재 섹션의 Docs URL
- `tasks`: 현재 섹션의 `###` 하위 작업 배열
각 `tasks` 항목은 다음 필드를 포함한다.
- `title`: `###` 하위 제목
- `steps`: numbered steps 배열
각 `steps` 항목은 다음 필드를 포함한다.
- `step_no`: 해당 task 안의 numbered step 번호
- `text`: numbered step 본문. Markdown의 step 문구에서 backtick만 제거한다.
- `fields`: numbered step 아래 bullet list의 field/value object. bullet list가 없으면 빈 object `{}`를 넣는다.
`설정값 입력값`처럼 code block으로 작성된 값 목록은 해당 section의 `tasks`에 `title`을 `설정값 입력값`으로 넣고, `steps` 첫 항목의 `fields`에 key/value object로 넣는다.
JSON에는 Markdown code fence, 설명 문장, 주석, trailing comma를 쓰지 않는다.
최종 결과에는 프롬프트 설명 없이 완성된 `.md` 본문과 `.json` 본문만 출력한다.
