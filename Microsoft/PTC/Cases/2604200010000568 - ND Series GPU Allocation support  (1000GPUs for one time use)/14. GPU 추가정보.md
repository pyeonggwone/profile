# GPU 추가 정보 및 답변 초안

## 추가 요청 요약

### 일정

- 희망 기간: 8월 1일 ~ 8월 30일
- 대체 가능 기간: 8월 확보가 어려운 경우 9월로 이전 가능

### 주요 요구 사항

- 동일 Azure region 내 다수의 GPU/CPU VM 인스턴스 필요
- 모든 VM 인스턴스는 고객사 아크릴의 커스텀 Kubernetes 환경에 편입되어 하나의 Kubernetes cluster로 운영되어야 함
- 총 GPU 수량은 256장 필요
- 256장 확보가 어려운 경우 128장까지 축소 가능
- 고스펙 GPU는 전체 GPU 수량의 약 50% 필요
- 고스펙 GPU는 B/H 계열 GPU를 우선 희망
- 고스펙 GPU 인스턴스 간 RDMA 지원 필수
- 저스펙 GPU는 전체 GPU 수량의 약 50% 필요
- CPU 인스턴스와 저사양 GPU 16장 규모 인스턴스는 1개월 동안 지속 사용 예정
- 대규모 GPU는 2주 동안 격일 테스트 방식으로 사용 예정이며, 총 사용일은 6일 이상

### 사용 계획

- 1~2주차: CPU 인스턴스와 소수의 저사양 GPU 인스턴스로 환경 구성
- 3~4주차: 격일로 대규모 GPU 테스트 수행
- 테스트일: GPU 256장 또는 128장 규모로 분산학습/분산추론 실행
- 비테스트일: CPU 인스턴스 및 저사양 GPU 환경에서 버그 수정, 환경 조정, 다음 테스트 준비

### 실행 대상 워크로드

- 유명 open-source LLM 대상 분산학습/분산추론
- 아크릴 보유 의료특화 파운데이션 모델 `A-LLM.H` 대상 분산학습/분산추론
- `JONATHAN GPUBASE` 적용 전후 성능 향상 정도 확인

## 검토 답변

이번 추가 정보 기준으로는 기존의 `1,000 GPU` 일회성 요청보다 범위가 현실적으로 정리되었습니다. 핵심은 한 달 전체 기간 동안 모든 GPU를 계속 점유하는 방식이 아니라, 기본 환경은 1개월 유지하고 대규모 GPU는 2주 동안 격일로 사용하는 burst 테스트 형태입니다.

고객 요구사항에 가장 부합하는 요청 방식은 아래와 같습니다.

- 동일 region 기준으로 capacity를 요청해야 합니다.
- 고스펙 GPU는 RDMA/InfiniBand가 필요한 분산학습 구간에 사용해야 합니다. H100/H200 계열은 기존 후보로 검토 가능하며, B 계열 GPU가 필요한 경우에는 현재 공개 후보 목록 기준의 정확한 Azure VM SKU와 region availability를 capacity team에 별도 확인해야 합니다.
- 저스펙 GPU와 CPU VM은 Kubernetes cluster의 일반 node pool 또는 보조 workload node로 구성하는 방향이 적합합니다.
- 고스펙 GPU와 저스펙 GPU를 하나의 Kubernetes cluster에 포함하는 것은 가능하지만, RDMA 성능이 필요한 학습 job은 RDMA 지원 GPU node group 안에서 실행되도록 분리해야 합니다.
- 256 GPU 전체를 같은 시점에 반드시 확보하기 어렵다면, 128 GPU 기준의 fallback 요청을 함께 제출하는 것이 좋습니다.
- 8월 capacity 확보가 어려울 경우 9월 수행 가능성을 명시하면 capacity team의 배정 옵션을 넓힐 수 있습니다.

## 권장 요청안

### 1순위: 256 GPU 기준

고스펙 GPU 128장과 저스펙 GPU 128장을 동일 region에서 확보하는 안입니다. 실제 사용 패턴상 저사양 GPU 16장은 1개월 상시 사용하고, 나머지 GPU capacity는 3~4주차 격일 테스트에 맞춰 burst 형태로 요청하는 방식이 적합합니다.

- 고스펙 GPU 후보: B 계열 GPU SKU 확인 필요, `Standard_ND96isr_H100_v5`, `Standard_ND96isr_H200_v5`, `Standard_ND96asr_v4`
- 고스펙 GPU 수량: 128 GPUs
- 8 GPU VM 기준 필요 수량: 16 VMs
- 필수 조건: GPU VM 간 RDMA/InfiniBand 지원
- 저스펙 GPU 후보: `NCas T4 v3-Series`, `NVadsA10_v5-Series`, `NCads A100 v4-Series` 중 region capacity에 맞춰 검토
- 저스펙 GPU 수량: 128 GPUs
- CPU VM: Kubernetes control plane, system workload, preprocessing, storage/monitoring 용도에 맞춰 별도 산정

이 안은 고객의 원 요구사항을 가장 잘 충족하지만, 동일 region에서 고스펙 GPU 128장과 저스펙 GPU 128장을 같은 기간에 확보해야 하므로 capacity 사전 확인이 필요합니다.

### 2순위: 128 GPU fallback 기준

256 GPU 확보가 어려울 때를 대비한 축소안입니다.

- 고스펙 GPU: 64 GPUs
- 8 GPU VM 기준 필요 수량: 8 VMs
- 저스펙 GPU: 64 GPUs
- CPU VM 및 저사양 GPU 16장 규모 환경은 1개월 유지
- 대규모 GPU는 2주 동안 격일 테스트로 사용

이 안은 전체 성능 검증 규모는 줄어들지만, `JONATHAN GPUBASE` 적용 전후 비교와 분산학습/분산추론 검증 목적에는 더 현실적인 capacity 요청안이 될 수 있습니다.

## 고객 회신 초안

안녕하세요,

공유해 주신 추가 정보를 기준으로 Azure GPU capacity 요청안을 다시 정리했습니다.

이번 요청은 8월 1일부터 8월 30일까지 동일 region 내 CPU VM, 저사양 GPU VM, 고성능 GPU VM을 함께 구성하고, 아크릴의 커스텀 Kubernetes 환경에서 하나의 cluster로 운영하는 형태로 이해했습니다. 8월 capacity 확보가 어려운 경우 9월 일정도 가능하다는 점은 capacity 검토 시 중요한 완화 조건으로 함께 반영하겠습니다.

기술적으로는 모든 VM을 동일 region과 동일 network boundary 안에 배치하여 Kubernetes node로 편입하는 방향이 적합합니다. 다만 RDMA/InfiniBand가 필요한 분산학습 workload는 RDMA를 지원하는 고성능 GPU node group 안에서 실행되도록 분리하는 것이 좋습니다. 저사양 GPU 또는 CPU VM은 같은 cluster에 포함할 수 있지만, 일반적으로 RDMA fabric 참여 대상은 아니므로 preprocessing, inference, bug fix, 테스트 준비, 보조 workload 용도로 분리하는 구성이 적합합니다.

GPU 수량은 우선 256장 기준으로 요청하되, capacity 확보가 어려운 경우 128장 fallback을 함께 제시하는 방식이 좋겠습니다. 256장 기준으로는 고성능 GPU 128장과 저사양 GPU 128장으로 나누어 검토하고, 고성능 GPU는 B 계열 GPU SKU 확인과 함께 H100/H200/A100 계열의 RDMA 지원 SKU를 우선 후보로 보겠습니다. 저사양 GPU는 T4/A10/A100 계열 중 동일 region에서 함께 확보 가능한 SKU를 기준으로 검토하는 것이 현실적입니다.

사용 패턴은 한 달 전체 기간 동안 256 GPU를 계속 사용하는 방식이 아니라, CPU 인스턴스와 저사양 GPU 16장 규모 환경은 1개월 유지하고, 대규모 GPU 240장 또는 112장 규모는 2주 동안 격일 테스트로 총 6일 이상 사용하는 형태로 정리하겠습니다. 이 정보는 capacity 요청 시 필요한 기간과 실제 점유 패턴을 설명하는 데 도움이 됩니다.

실행 workload는 open-source LLM과 아크릴 의료특화 파운데이션 모델 `A-LLM.H`의 분산학습/분산추론이며, 최종 목적은 `JONATHAN GPUBASE` 적용에 따른 성능 향상 검증으로 이해했습니다.

위 기준으로 Microsoft 내부 capacity 검토 또는 Azure Support quota/capacity 요청 시 사용할 수 있는 요청 문구를 함께 준비하겠습니다.
