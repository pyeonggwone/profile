---
`2. Portal Step Guide.md`와 `3. Solution.md`를 읽고 `4. Final.md`를 작성한다.
목적은 새 답변 작성이 아니라 `3. Solution.md`의 질문별 답변, 설명, 주의, reference를 유지한 채 구현 절차를 덧붙이는 것이다.
`3. Solution.md`의 문단형 설명을 삭제하거나 표로 축약하지 않는다.
표는 비교, 목록, 입력값처럼 표가 읽기 쉬운 경우에만 사용하고, 설명형 답변은 문단과 bullet로 작성한다.
결과물에서 `2. Portal Step Guide.md`, `3. Solution.md` 같은 내부 파일명을 노출하지 않는다.
각 질문 섹션은 `답변`을 먼저 쓰고 이어서 `구현 방법`, `주의`, `reference`를 자연스럽게 연결한다.
`구현 방법`은 관련 portal menu path, object type, sample value만 간결하게 포함한다.
`확인`이라는 표현은 실제 portal 화면에서 사용자가 볼 항목에만 제한하고 별도 확인 전용 섹션을 만들지 않는다.
sample tenant ID, domain, user, group, role, policy 이름은 예시값으로 표시한다.
production 변경 가능성이 있는 항목은 `pilot` 또는 `report-only` 맥락으로 표현한다.
Portal URL, Docs URL, 전체 numbered steps는 본문에 복사하지 않는다.
solution에 없는 새 제품, 기능, architecture 판단을 추가하지 않는다.
기존 `2. Portal Step Guide.md`와 `3. Solution.md`는 삭제하거나 대폭 수정하지 않는다.
reference는 `3. Solution.md`의 링크를 유지하고 필요한 구현 항목에만 연결한다.
마지막에는 전체 pilot 흐름을 7단계 이내로 요약한다.
마지막에는 sample value를 실제 고객값으로 바꾸기 위한 입력값만 정리한다.
결과는 고객이 읽을 수 있는 한국어 Markdown 본문만 포함한다.
