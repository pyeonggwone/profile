두 요구사항 정리
대체 가능한 부분은 대체하되, 직접 수정 가능한 것만 (블랙박스 X)
업로드 ↔ 다운로드 양방향 (PDF 읽기 + PDF 쓰기 둘 다)
요구사항 1: "대체하지만 수정 가능"의 의미
= 소스 공개된 표준 구현체만 쓰고, 블랙박스 SaaS/상용 SDK는 금지

분류	대체 가능?	이유
OS 표준 zlib (소스 공개)	✅ 가능	RFC 1950 표준, 소스 수정 가능
OS 표준 JPEG 디코더 (libjpeg)	✅ 가능	ISO 표준, 소스 공개
HTTP 서버 표준 라이브러리	✅ 가능	언어 표준, 수정 가능
Adobe PDF Library (상용)	❌ 불가	블랙박스, 수정 불가
클라우드 PDF API	❌ 불가	외부 의존, 수정 불가
→ 즉 "표준 + 소스 공개"인 것만 대체로 인정, 그 외는 직접 구현.

요구사항 2: 양방향 (PDF ↔ 웹)
Code
방향 A (업로드):
사용자 PDF → 서버 분석 → 웹 표시

방향 B (다운로드):
웹에서 편집/생성 → 서버가 PDF 만들기 → 사용자 다운로드
→ PDF 읽기 + PDF 쓰기 둘 다 직접 구현 필요.

대체 vs 직접 구현 결정표
✅ 대체 OK (소스 공개 표준 구현체)
부품	대체할 것	이유
HTTP 서버	OS/언어 표준 라이브러리	표준, 수정 가능
TCP 소켓	OS 시스템 콜	표준
파일 I/O	OS 시스템 콜	표준
zlib (Flate)	OS의 zlib (RFC 1950 표준 구현)	표준, 소스 수정 가능
JPEG (DCT)	libjpeg (ISO 10918 표준 구현)	표준, 소스 수정 가능
JPEG2000 (JPX)	libopenjp2 (ISO 15444 표준 구현)	표준, 소스 수정 가능
암호화 (AES)	OS 표준 (OpenSSL 등)	표준, 소스 수정 가능
JSON 직렬화	언어 표준 라이브러리	표준
SQLite	표준 DB	소스 공개

부품	이유
PDF 파서/생성기	PDF 고유 문법, 직접 구현해야 100% 제어

RLE 디코더/인코더	매우 단순, 직접

양방향 구현 시 주의점
방향 A (읽기) → 방향 B (쓰기) 라운드트립
원본 PDF를 읽어서 → 편집 → 다시 PDF로 저장할 때 데이터 손실 없어야 함.

이를 위해:

읽을 때: 모든 객체를 메모리에 보존 (모르는 객체도 그대로 유지)
편집 시: 변경된 객체만 표시
쓸 때:
수정된 객체는 새 버전으로
건드리지 않은 객체는 원본 바이트 그대로 (재인코딩 X)
Incremental Update 방식 사용 (PDF 표준 기능, 원본 끝에 변경분만 추가)
→ 이게 100% 호환의 핵심. 모든 객체를 일일이 다시 만들면 미묘한 차이로 깨질 수 있음. 수정 안 한 부분은 원본 바이트 보존.

Incremental Update 구조
Code
[원본 PDF 전체 바이트]                ← 그대로 유지
[새 객체들]                           ← 수정/추가된 것만
[새 xref 테이블]
[새 trailer (이전 trailer 가리킴)]
%%EOF
→ 원본 무손실 + 수정 가능 + PDF 표준 100% 호환.

추천 진행 순서 (양방향)
Phase 1: 양방향 뼈대
HTTP 서버 (업로드 + 다운로드 엔드포인트)
파일 저장/전송
Phase 2: PDF 읽기 파서
xref → 객체 → stream
Filter 디코더 (대체 OK인 건 OS 호출, 나머지 직접)
Phase 3: PDF 쓰기 생성기
객체 → 직렬화
stream 인코더 (대체 OK인 건 OS 호출, 나머지 직접)
xref/trailer 작성
Phase 4: 라운드트립 검증
원본 PDF 읽기 → 변경 없이 다시 쓰기 → 바이트 일치 또는 의미적 일치 확인
Phase 5: 분석 기능
텍스트/이미지/메타데이터 추출
Phase 6: 편집 기능
텍스트/이미지/주석 추가
Incremental Update로 저장
Phase 7: 웹 표시 (Canvas 직접 렌더링)
PDF 명령어 → Canvas 2D API
"100% 호환"을 위한 추가 원칙
모르는 Filter 만나면 에러 X, 원본 바이트 보존
모르는 객체 타입 만나면 그대로 유지
PDF 1.0 ~ 2.0 모든 버전 지원
암호화 PDF는 Crypt Filter 처리 후 동일 흐름
선형화 (Linearized) PDF 구조 인식
객체 스트림 (PDF 1.5+) 지원
Cross-reference Stream (PDF 1.5+) 지원
→ 이걸 다 구현하면 모든 PDF 100% 호환.

정리
✅ 대체 가능 (소스 공개 표준 구현체): zlib, libjpeg, libopenjp2, OS 표준 라이브러리, SQLite ❌ 직접 구현 필수: PDF 파서/생성기, 폰트 처리, 그리기 명령 해석, 단순 Filter (LZW/RLE/Hex/85), 복잡한 Filter (CCITT/JBIG2)

양방향 핵심:

읽기 = 모든 객체 보존하며 파싱
쓰기 = Incremental Update로 원본 무손실 보존
100% 호환 = "모르는 부분은 건드리지 말고 그대로 유지"
AI한테 시킬 때 단계별로 쪼개고, 매 단계 끝에 라운드트립 테스트 (읽고 다시 쓴 게 원본과 동일한가) 넣으면 호환성 보장됨.
