## 1. 이 문서를 왜 작성했는가

`README copy 2.md`는 PDF를 직접 분석하거나 PDF 기반 웹 도구를 만들기 위해 필요한 개념을 한 번에 정리한 문서다.

처음에는 PDF 압축 방식, zlib, qpdf, PDF.js, 파일 업로드, 직접 파서 구현 같은 내용이 흩어져 있었다. 그래서 누군가에게 설명할 때 바로 사용할 수 있도록 다음 흐름으로 재구성했다.

1. PDF 내부가 어떻게 생겼는지 설명한다.
2. PDF 내부 데이터가 왜 깨진 바이너리처럼 보이는지 설명한다.
3. 그 바이너리를 풀기 위해 어떤 Filter와 디코더가 필요한지 설명한다.
4. 그중 가장 중요한 `/FlateDecode`와 zlib/Deflate를 설명한다.
5. 웹 서비스로 만들 때 어떤 구조가 필요한지 설명한다.
6. 직접 구현한다면 어떤 순서로 만들어야 하는지 설명한다.

즉, 이 문서는 단순한 PDF 지식 목록이 아니라 “PDF 분석기를 직접 만들려면 어떤 순서로 생각해야 하는가”를 보여주기 위한 문서다.

## 2. 설명의 출발점: PDF는 단순 텍스트 파일이 아니다

누군가에게 설명할 때 가장 먼저 잡아야 할 관점은 이것이다.

PDF는 겉으로 보면 문서 파일이지만, 내부적으로는 객체, 사전, 배열, stream, 압축 데이터, 이미지, 폰트, 메타데이터가 섞인 복합 파일 포맷이다.

그래서 PDF 파일을 텍스트 에디터로 열면 일부는 읽히지만, 많은 부분은 깨진 문자처럼 보인다. 그 이유는 PDF 안의 중요한 데이터가 압축된 stream으로 들어가기 때문이다.

이 지점을 먼저 설명해야 이후의 Filter, zlib, 디코더, 파서 이야기가 자연스럽게 이어진다.

## 3. 첫 번째 핵심: PDF 내부 데이터는 stream에 들어간다

PDF에서 실제 페이지 내용, 텍스트 그리기 명령, 이미지, 폰트, 메타데이터 등은 `stream ... endstream` 사이에 저장될 수 있다.

예를 들어 다음 stream은 페이지에 `Hello`라는 텍스트를 그리는 PDF 명령을 담고 있다.

```pdf
4 0 obj
<</Length 50>>
stream
BT /F1 12 Tf 100 700 Td (Hello) Tj ET
endstream
endobj
```

이 예시를 먼저 보여주는 이유는 PDF가 최종적으로 “문자열을 저장하는 파일”이 아니라 “명령과 리소스를 저장하는 파일”이라는 점을 설명하기 위해서다.

여기서 `BT`, `Tf`, `Td`, `Tj`, `ET` 같은 명령은 PDF 렌더러가 텍스트를 그릴 때 해석하는 명령이다. 따라서 PDF에서 텍스트를 추출하려면 단순히 문자열 검색만 하면 안 되고, stream 안의 PDF 명령을 해석해야 한다.

## 4. 두 번째 핵심: stream은 Filter로 압축되거나 인코딩된다

PDF stream은 그대로 저장될 수도 있지만, 대부분은 용량을 줄이기 위해 압축된다. 이때 PDF는 stream 사전에 `/Filter` 항목을 넣어 어떤 방식으로 처리되었는지 표시한다.

예를 들어 다음 구조는 stream이 zlib/Deflate로 압축되어 있다는 뜻이다.

```pdf
4 0 obj
<</Length 24/Filter/FlateDecode>>
stream
[압축된 바이너리 바이트]
endstream
endobj
```

이 부분을 설명하는 의도는 분명하다.

PDF를 분석하려면 먼저 stream을 찾아야 하고, 그 다음 `/Filter`를 확인해야 하며, 마지막으로 해당 Filter에 맞는 디코더로 데이터를 풀어야 한다.

즉, PDF 분석의 기본 순서는 다음과 같다.

1. 객체를 찾는다.
2. stream을 찾는다.
3. `/Filter`를 확인한다.
4. Filter에 맞게 디코드한다.
5. 디코드된 내용을 분석한다.

## 5. 세 번째 핵심: 대부분의 PDF stream은 FlateDecode가 중요하다

PDF에는 여러 Filter가 있지만, 실무에서 가장 먼저 이해해야 할 것은 `/FlateDecode`다.

`/FlateDecode`는 zlib/Deflate 압축을 의미하며, 텍스트, 폰트, 벡터 그래픽, 메타데이터 등 여러 종류의 stream에 광범위하게 사용된다.

다른 Filter도 중요하지만 우선순위는 다르다.

| 우선순위 | Filter | 이유 |
| --- | --- | --- |
| 높음 | `/FlateDecode` | 일반 PDF stream에서 가장 자주 사용됨 |
| 높음 | `/DCTDecode` | JPEG 이미지가 PDF에 포함될 때 사용됨 |
| 중간 | `/ASCIIHexDecode`, `/ASCII85Decode`, `/RunLengthDecode` | 구현은 비교적 단순하지만 사용 빈도는 낮음 |
| 중간 | `/LZWDecode` | 오래된 PDF에서 볼 수 있음 |
| 낮음 또는 후순위 | `/CCITTFaxDecode`, `/JBIG2Decode`, `/JPXDecode` | 스캔 문서나 특수 이미지에서 사용됨 |

이 순서로 설명한 이유는 처음부터 모든 Filter를 동일한 비중으로 다루면 듣는 사람이 무엇부터 구현해야 하는지 놓치기 쉽기 때문이다.

## 6. 네 번째 핵심: zlib과 Deflate를 이해해야 한다

`/FlateDecode`를 이해하려면 zlib과 Deflate를 구분해서 설명해야 한다.

Deflate는 압축 알고리즘이고, zlib은 그 알고리즘을 사용하는 데이터 형식과 구현체로 이해하면 된다. Deflate는 크게 LZ77과 Huffman 부호화를 조합한다.

1. LZ77은 앞에서 나온 반복 패턴을 찾아 짧게 표현한다.
2. Huffman 부호화는 자주 나오는 값에 짧은 비트를 배정한다.

이 설명을 넣은 이유는 PDF stream 안의 깨진 바이너리가 아무 의미 없는 데이터가 아니라, 규칙에 따라 압축된 결과라는 점을 보여주기 위해서다.

하지만 실제 구현에서는 Deflate를 직접 손으로 구현하는 것이 목적이 아니다. 대부분의 경우 운영체제나 언어에서 제공하는 zlib API를 호출하면 된다.

따라서 설명할 때는 다음 메시지가 중요하다.

```text
PDF 파서를 직접 만들더라도 zlib 알고리즘 자체를 직접 구현할 필요는 없다.
대신 PDF 구조를 파싱하고, /FlateDecode stream을 찾아 zlib으로 풀 수 있으면 된다.
```

## 7. 다섯 번째 핵심: 웹에서 PDF를 처리하려면 디코더가 필요하다

웹에서 PDF를 보여주는 대표적인 도구는 PDF.js다. PDF.js는 단순히 PDF 파일을 화면에 띄우는 라이브러리가 아니라, PDF 내부 구조를 읽고 여러 Filter를 해제하고 페이지 명령을 해석하는 렌더링 엔진이다.

그래서 PDF.js 없이 직접 구현하려면 다음 요소들이 필요하다.

| 필요한 요소 | 역할 |
| --- | --- |
| PDF 구조 파서 | 객체, xref, trailer, Catalog, Pages 구조를 읽음 |
| stream 디코더 | `/FlateDecode`, `/DCTDecode`, `/ASCIIHexDecode` 등을 해제 |
| 텍스트 분석기 | `Tj`, `TJ`, ToUnicode CMap 등을 해석 |
| 이미지 추출기 | XObject Image stream을 추출 |
| 렌더러 | PDF 그래픽 명령을 Canvas 등에 그림 |

이 내용을 넣은 이유는 “웹에서 PDF 업로드 후 분석”이라는 요구가 단순 파일 업로드 문제가 아니라 PDF 엔진 구현 문제로 이어진다는 점을 설명하기 위해서다.

## 8. 여섯 번째 핵심: PDF 업로드 서비스는 파싱, 디코딩, 분석으로 나뉜다

웹 기반 PDF 분석 도구를 설명할 때는 전체 흐름을 먼저 보여주는 것이 좋다.

1. 사용자가 PDF를 업로드한다.
2. 서버가 PDF 파일을 저장한다.
3. 서버가 PDF 파일 구조를 파싱한다.
4. stream의 `/Filter`를 확인한다.
5. Filter에 맞는 디코더로 stream을 해제한다.
6. 해제된 stream에서 텍스트, 이미지, 메타데이터 등을 추출한다.
7. 결과를 JSON이나 SQLite에 저장한다.
8. 브라우저에서 원본 PDF와 분석 결과를 보여준다.

이 순서로 설명하면 프론트엔드, 서버, PDF 파서, 분석 결과 저장이 각각 어떤 역할을 맡는지 분리해서 이해할 수 있다.

## 9. 일곱 번째 핵심: 직접 구현할 때는 PDF 구조부터 읽어야 한다

PDF 분석을 직접 구현할 때 가장 먼저 해야 할 일은 stream 디코드가 아니라 파일 구조 파싱이다.

PDF에는 객체가 여러 개 있고, 각 객체의 위치를 찾기 위해 `xref`와 `trailer`를 읽어야 한다. 파일 끝의 `startxref`를 찾고, 그 위치에서 xref 테이블을 읽은 뒤, trailer에서 Root Catalog를 찾아야 문서 구조를 따라갈 수 있다.

그래서 구현 순서는 다음처럼 잡았다.

1. `startxref` 찾기
2. xref 테이블 읽기
3. 객체 번호와 바이트 오프셋 매핑
4. trailer 파싱
5. Catalog 찾기
6. Pages 트리 순회
7. Page의 Contents stream 찾기
8. stream Filter 해제
9. 텍스트, 이미지, 폰트, 양식, 주석 분석

이 순서를 넣은 이유는 PDF 분석에서 가장 흔한 착각을 피하기 위해서다.

```text
PDF 분석은 압축 해제부터 시작하는 것이 아니라,
어떤 객체가 어디에 있고 어떤 stream을 풀어야 하는지 찾는 것부터 시작한다.
```

## 10. 단계별 구현 계획을 넣은 이유

`README copy 2.md`에 Phase 1부터 Phase 7까지 구현 계획을 넣은 이유는 실제 개발 작업으로 옮길 수 있도록 하기 위해서다.

개념 설명만 있으면 “무엇을 알아야 하는가”는 알 수 있지만, “무엇부터 만들면 되는가”는 불명확할 수 있다. 그래서 다음 순서로 나눴다.

| Phase | 목표 | 이유 |
| --- | --- | --- |
| Phase 1 | 파일 업로드 뼈대 | PDF 분석 전에 입력과 저장 흐름을 확보 |
| Phase 2 | PDF 구조 파싱 | 객체 위치와 문서 구조를 알아야 stream을 찾을 수 있음 |
| Phase 3 | Stream 디코드 | 실제 데이터가 압축되어 있으므로 해제 필요 |
| Phase 4 | 분석 기능 | 텍스트, 이미지, 메타데이터 등 목적 데이터 추출 |
| Phase 5 | 결과 저장 | 분석 결과를 재사용 가능하게 저장 |
| Phase 6 | 웹 표시 | 사용자에게 원본과 분석 결과를 제공 |
| Phase 7 | 직접 렌더링 엔진 | PDF.js 없이 직접 표시하려는 경우의 확장 단계 |

이 구조는 프로젝트를 작게 시작해서 점진적으로 확장하기 위한 순서다.

## 11. 관련 표준을 정리한 이유

PDF, HTTP 업로드, zlib, JPEG, JSON 등은 모두 별도 표준에 기반한다.

관련 표준을 정리한 이유는 구현 중 막혔을 때 참고해야 할 원문 기준을 명확히 하기 위해서다.

| 영역 | 참고 표준 |
| --- | --- |
| PDF 구조 | ISO 32000-2 |
| HTTP | RFC 9110 |
| 파일 업로드 | RFC 7578 |
| zlib | RFC 1950 |
| Deflate | RFC 1951 |
| JPEG | ISO/IEC 10918 |
| JPEG 2000 | ISO/IEC 15444 |
| JBIG2 | ISO/IEC 14492 |
| JSON | RFC 8259 |

이 표준 목록은 문서의 신뢰도를 높이기 위한 목적도 있지만, 더 중요한 목적은 “직접 구현할 때 어디까지가 내가 만들어야 할 부분이고 어디부터가 표준 구현을 호출하면 되는 부분인지”를 구분하는 것이다.

## 12. AI에게 작업을 나눠 요청하는 방식

PDF 파서 전체를 한 번에 만들라고 요청하면 범위가 너무 커진다. 그래서 작은 단위로 나눠서 요청하는 방식이 필요하다.

좋은 요청 단위는 다음과 같다.

1. 파일 업로드 서버 만들기
2. `startxref` 찾는 함수 만들기
3. xref 테이블 파싱 함수 만들기
4. PDF Dictionary 파서 만들기
5. `/FlateDecode` stream 해제 함수 만들기
6. `/ASCIIHexDecode` 해제 함수 만들기
7. Contents stream에서 `Tj`, `TJ` 명령 찾기
8. 분석 결과를 JSON으로 저장하기

이렇게 나누면 각 기능을 독립적으로 테스트할 수 있고, 문제가 생겼을 때 어느 단계에서 실패했는지 찾기 쉽다.

AI에게 요청할 때는 다음처럼 입력과 출력을 명확히 지정하는 것이 좋다.

```text
PDF의 xref 테이블을 파싱하는 함수를 작성해줘.
입력은 PDF 전체 bytes이고, 출력은 객체 번호와 바이트 오프셋을 담은 dict로 해줘.
```

```text
PDF stream의 /ASCIIHexDecode Filter를 해제하는 함수를 작성해줘.
입력은 bytes, 출력도 bytes로 해줘.
잘못된 hex 문자가 있으면 예외를 발생시켜줘.
```

## 13. 전체 설명 순서 요약

누군가에게 이 내용을 설명할 때는 다음 순서로 말하면 된다.

1. PDF는 단순 문서가 아니라 객체와 stream으로 구성된 복합 포맷이다.
2. 실제 페이지 내용과 리소스는 stream 안에 들어간다.
3. stream은 대부분 압축되어 있어서 텍스트 에디터로 보면 깨져 보인다.
4. PDF는 `/Filter`로 어떤 압축이나 인코딩을 사용했는지 표시한다.
5. 가장 중요한 Filter는 `/FlateDecode`이고, 이것은 zlib/Deflate 압축이다.
6. Deflate는 LZ77과 Huffman 부호화를 조합한 압축 알고리즘이다.
7. 직접 구현할 때 Deflate 알고리즘 자체를 만들 필요는 없고 zlib API를 호출하면 된다.
8. 웹에서 PDF를 분석하려면 업로드, 저장, PDF 구조 파싱, stream 디코드, 결과 추출, 결과 표시가 필요하다.
9. 직접 PDF 파서를 만들려면 `startxref`, xref, trailer, Catalog, Pages 순서로 문서 구조를 따라가야 한다.
10. 그 다음 Contents stream, 이미지 XObject, Font, AcroForm, Annots 같은 분석 대상을 처리한다.
11. 구현은 Phase 단위로 쪼개야 하고, AI에게도 함수 단위로 작게 요청하는 것이 좋다.

결론적으로 `README copy 2.md`는 PDF 자체를 설명하려는 문서이면서, 동시에 PDF 분석 도구를 직접 만들기 위한 개발 로드맵이다. `README copy 3.md`는 그 문서를 누군가에게 설명할 때 어떤 의도와 순서로 말하면 되는지를 정리한 설명용 문서다.
